# sp-9xr.11 — Terminal flow-control: wedge metric + backpressure confirmation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Observe when the browser terminal is at risk of wedging (xterm.js parses ~5–35 MB/s with a ~50 MB write-buffer cap), via a **wedge metric** on TerminalView's incoming-byte backlog. Backpressure itself already exists end-to-end (synchronous blocking sends node→CP→browser); a full ACK/credit scheme is intentionally punted to post-launch (decide from the metric's real data).

**Architecture:** TerminalView writes received bytes with `term.write(data, callback)` — the callback fires when xterm has parsed the chunk. Track outstanding (written-but-not-yet-parsed) bytes; each time the backlog crosses a threshold (rising edge) count a "wedge" and `console.warn`. Extract the counting into a tiny pure `BacklogTracker` (unit-tested). Document the node→CP→browser backpressure chain so it's clear why no credit scheme is needed yet.

**Tech Stack:** web TypeScript + vitest; a one-line node/relay doc comment. **Toolchain:** `cd web && npx vitest run`; Go (doc only) `export PATH="/var/home/alepar/sdk/go1.26.0/bin:$PATH"`.

### Context (verified)
- `web/src/views/TerminalView.tsx` (`.7`): `onMessage: (data) => term.write(typeof data === "string" ? data : new Uint8Array(data))`. xterm's `Terminal.write(data, callback?)` invokes `callback` after the chunk is processed — the hook for backlog tracking.
- Backpressure chain (all synchronous-blocking, so a slow browser backpressures the agent): node `tmuxRelay` reader → `r.send` → node→CP gRPC `stream.Send` (flow-controlled) → CP `runNode` receive → `router.FromNode` → `wsClient.Send` → `conn.Write` (coder/websocket, blocks on a slow browser) → browser ws → `term.write`. No unbounded buffer in the path; the `tmuxRelay` read loop pauses when `send` blocks, which stalls the PTY and thus the agent.

```ts
// BacklogTracker counts how often the unparsed-byte backlog crosses a threshold (a proxy for the
// browser terminal "wedging" — xterm.js parses ~5–35 MB/s with a ~50 MB write-buffer cap). It does
// NOT throttle; it only observes. A full ACK/credit scheme is deferred until this metric shows it's
// needed (sp-9xr.11 punt). onWrite when bytes are handed to term.write; onAck in term.write's callback.
export class BacklogTracker {
  outstanding = 0;
  wedges = 0;
  private over = false;
  constructor(private threshold: number) {}
  onWrite(n: number) {
    this.outstanding += n;
    if (!this.over && this.outstanding >= this.threshold) {
      this.over = true;
      this.wedges++;
    }
  }
  onAck(n: number) {
    this.outstanding = Math.max(0, this.outstanding - n);
    if (this.over && this.outstanding < this.threshold) this.over = false;
  }
}
```

```ts
const backlog = new BacklogTracker(8 * 1024 * 1024); // 8 MB — well under xterm's ~50 MB cap

onMessage: (data) => {
  const bytes = typeof data === "string" ? new TextEncoder().encode(data) : new Uint8Array(data);
  const n = bytes.length;
  backlog.onWrite(n);
  const before = backlog.wedges;
  term.write(bytes, () => backlog.onAck(n));
  if (backlog.wedges > before) {
    console.warn(`terminal backlog wedge: spawn=${spawnId} outstanding=${backlog.outstanding}B wedges=${backlog.wedges}`);

},
```
