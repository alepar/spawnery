> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A tmux-mode spawn launches its agent TUI inside a detached tmux session in the container (and stays alive), so clients can later `tmux attach`. This is the container/node-side launch; the web/spawnctl attach transports are `.6`/`.9`.

**Architecture:** Make the agent image's `entrypoint.sh` argv-aware (run a passed command instead of the default opencode path — this fixes the `Cmd`-vs-`ENTRYPOINT` divergence, `sp-9xr.3.8`). Add a baked `spawn-tmux` wrapper that does `tmux new-session -d -s spawn -- <argv>` then blocks to keep PID1 alive. The manager, for tmux mode, sets `AgentSpec.Cmd = ["spawn-tmux", <runnable.Launch...>]` (replacing the tmux-reject). Add an `opencode-tui` runnable (mode tmux) so there's a tmux runnable backed by the real tmux-capable image for verification. **The CP-side tmux gate stays up** (`internal/cp/server.go`) so this is dormant via normal flow until `.8`; verified directly here.

**Tech Stack:** Docker image (debian + tmux + opencode), Go (`spawnlet` manager, `agentcaps`), stdlib `testing`. **Verifiable here** (Docker 29.4.2 + built images).

## Task 1: Argv-aware entrypoint + tmux wrapper (image)

**Files:**
- Modify: `deploy/agent/entrypoint.sh`
- Create: `deploy/agent/spawn-tmux`
- Modify: `deploy/agent/Dockerfile`

- [ ] **Step 1: Make entrypoint.sh argv-aware**

In `deploy/agent/entrypoint.sh`, immediately after the `set -e` line, add:

```sh
# Argv-aware: if the node passes an explicit command (e.g. the tmux launcher for tmux-mode
# runnables), run it INSTEAD of the default opencode-serve + adapter path. This makes a
# container-command override actually take effect under the tini ENTRYPOINT.
if [ "$#" -gt 0 ]; then
  exec "$@"
fi
```

- [ ] **Step 2: Add the tmux launch wrapper**

Create `deploy/agent/spawn-tmux`:

```sh
#!/bin/sh
# Launch the agent TUI inside a detached tmux session named "spawn", then block so the container
# (PID 1) stays alive while clients attach via `tmux attach -t spawn`. Args are the agent argv.
set -e
SESSION="${SPAWN_TMUX_SESSION:-spawn}"
export TERM="${TERM:-xterm-256color}"
tmux new-session -d -s "$SESSION" -- "$@"
# Keep PID 1 alive for the life of the tmux session; exit when the server (and thus the session) ends.
exec tmux wait-for "spawnery-tmux-${SESSION}" 2>/dev/null || exec tail -f /dev/null
```

(`tmux wait-for` blocks indefinitely on a channel that's never signaled, keeping PID 1 alive; the `|| tail -f` is a portability fallback. When the container is stopped, tmux dies with it.)

- [ ] **Step 3: Bake it into the image**

In `deploy/agent/Dockerfile`, alongside the existing `COPY deploy/agent/spawn-tui.sh /usr/local/bin/spawn-tui` line, add:

```dockerfile
COPY deploy/agent/spawn-tmux /usr/local/bin/spawn-tmux
```

and add `/usr/local/bin/spawn-tmux` to the existing `chmod +x` line:

```dockerfile
RUN chmod +x /entrypoint.sh /usr/local/bin/spawn-tui /usr/local/bin/spawn-tmux
```

- [ ] **Step 4: Build the image + verify the tmux launch (real Docker)**

Run (from repo root; takes a few minutes):
```bash
docker build -t spawnery/agent:dev -f deploy/agent/Dockerfile .
cid=$(docker run -d --entrypoint /usr/bin/tini spawnery/agent:dev -- /entrypoint.sh /usr/local/bin/spawn-tmux opencode)
sleep 3
docker exec "$cid" tmux list-sessions
docker exec "$cid" tmux capture-pane -p -t spawn | head -20
docker rm -f "$cid"
```
Expected: `tmux list-sessions` shows a `spawn` session; `capture-pane` returns terminal content (opencode TUI output/startup, even if it errors without the sidecar — the point is the tmux session launched and hosts the process). This proves argv-passthrough + the tmux wrapper work in the real image.

