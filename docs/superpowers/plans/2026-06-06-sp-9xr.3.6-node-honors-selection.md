# sp-9xr.3.6 — Node honors the selected image + runnable launch (reject tmux) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** When `StartSpawn` carries an agent selection, the node launches the **selected image** (not the node's configured default) and runs the **runnable's launch argv** as the agent container command; tmux-mode is rejected at the node (defense in depth — launchable in Phase 2). Empty selection = today's behavior exactly.

**Architecture:** Add `agentcaps.FindRunnable(id)` to resolve a runnable globally. Add `Cmd []string` to `runtime.AgentSpec` and pass it through both backends (docker — `ContainerSpec.Cmd` already maps to the container config; CRI — `ContainerConfig.Command`). Keep `Manager.Create` as a thin wrapper and add `Manager.CreateWithSelection` (the only path that overrides image/cmd and rejects tmux); the node calls it. This keeps all ~16 existing `Create` callers untouched.

**Verification caveat:** real container launch is exercised only by the **Docker-gated e2e** (not run in this environment). The new selection LOGIC (selected image, launch argv, tmux reject) is fully unit-tested via a fake `PodBackend` that captures the `AgentSpec`. The backend `Cmd` plumbing itself is one-liners verified by `go build` + the existing e2e.

**Tech Stack:** Go (`spawnery`), `internal/runtime` (+`cri`), `internal/spawnlet`, `internal/agentcaps`, stdlib `testing`.

```go
// FindRunnable resolves a runnable by its id across all binaries. Runnable ids are globally
// unique (see TestRunnableIDsGloballyUnique), so the first match is unambiguous.
func FindRunnable(id string) (Runnable, bool) {
	for _, rs := range registry {
		for _, r := range rs {
			if r.ID == id {
				return r, true
			}
		}
	}
	return Runnable{}, false
}
```
