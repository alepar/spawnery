# sp-9xr.16 — goose-acp: bridge goose stdio-ACP to the node's TCP ACP listener Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Wire the `goose-acp` runnable so it works end-to-end. goose speaks ACP over **stdio** (`goose acp`), but the node attaches ACP over **TCP** (`ACP_LISTEN=tcp://0.0.0.0:7000`, dialed by the node's `Pump`). So the dispatcher needs a tiny **stdio↔TCP bridge**: listen on the node's TCP port, accept its single long-lived connection, spawn `goose acp` with its stdin/stdout wired to that connection, relay bytes raw, and loop on disconnect. Then `goose-acp → node Pump fanout → web ChatView` works (same canonical-ACP path opencode-served uses, but goose is the agent and the node Pump parses the frames).

**Architecture:** Add a small generic Go bridge `deploy/agent/acpexec` that mirrors `acpadapter`'s proven shape — reuse the **same `ACP_LISTEN` parsing** (`tcp://host:port` | `unix:<path>`) and the **one-connection-at-a-time** accept loop. The difference: instead of translating to opencode over HTTP, `acpexec` runs an arbitrary stdio-ACP agent argv (passed as its own args, e.g. `acpexec goose acp`) and pipes the accepted connection ↔ the child's stdin/stdout. On the node's disconnect it kills the child and loops to accept the next connection (fresh agent process — non-lossless, consistent with the rest of the system). The dispatcher's `goose-acp` case sets goose's OpenAI-sidecar env (identical to `goose-tui`) then `exec /usr/local/bin/acpexec goose acp`. No CP/node/agentcaps/manager Go changes — the manager already sets `ACP_LISTEN` for every agent and passes `Cmd=[runnableID]`; the node already runs the Pump for `ModeACP`.

**Tech Stack:** Go (new `deploy/agent/acpexec`, built in the Dockerfile gobuild stage), Docker image, real Docker + OpenRouter for the e2e. **Run Go:** `export PATH="/var/home/alepar/sdk/go1.26.0/bin:$PATH"`.

## Task 1: The `acpexec` stdio↔TCP bridge

**Files:** `deploy/agent/acpexec/main.go`, `deploy/agent/acpexec/main_test.go`

- [ ] **Step 1: Verify goose ACP (the unknown above).** Build the current image if needed (`docker build -t spawnery/agent:dev -f deploy/agent/Dockerfile .`) and run the `goose acp --help` check. Record what you find in the task report. Decide the exact agent argv (`goose acp` unless the check says otherwise).

- [ ] **Step 2: Write the bridge.** `deploy/agent/acpexec/main.go` — a generic stdio-ACP→TCP relay. Reuse `acpadapter`'s `listenSpec()`/`envOr()` shape verbatim (copy them; this is a separate `main` package — small duplication is fine and keeps the agent tools independent). Behavior:
  - `os.Args[1:]` is the agent argv (e.g. `["goose","acp"]`); fatal if empty.
  - Resolve `(network, addr)` from `ACP_LISTEN` (same parser as acpadapter), `net.Listen`, log `"acpexec: listening on %s %s, agent %v"`.
  - Loop: `conn := ln.Accept()`; start `exec.CommandContext(ctx, argv[0], argv[1:]...)` with `cmd.Stdin = conn`, `cmd.Stdout = conn`, `cmd.Stderr = os.Stderr`, inheriting the process env (`cmd.Env = os.Environ()` is the default — goose env is already exported by the dispatcher). `cmd.Start()`.
  - Relay is implicit (stdin/stdout are the conn). Wait for **either** the child to exit **or** the connection to close, then ensure both are torn down: when `conn.Read` returns EOF the child's stdin closes (most agents exit); also `cmd.Wait()` in a goroutine and, on the node disconnecting first, kill the child (`cmd.Process.Kill()`) so a fresh one starts next accept. Make sure there are **no zombie children** (always `cmd.Wait()` — this was a real bug caught in `.6`/the tmux relay; do not repeat it).
  - `_ = conn.Close()`; loop to accept the next node connection.
  - Implementation note: because `cmd.Stdin = conn` and `cmd.Stdout = conn`, Go wires them directly — no manual `io.Copy` needed. To detect the node closing the socket while the child is still alive, set up a small watcher: a goroutine that does `cmd.Wait()` and signals a channel; the main path also needs to notice a dead conn. Simplest robust shape: run `cmd.Wait()` in a goroutine → `done` channel; separately, since `conn` is wired to the child's stdio, the child exits when the node closes the conn (stdin EOF) — so awaiting `done` and then `conn.Close()` covers the normal case. Add a `defer cmd.Process.Kill()` (ignored error if already exited) as the safety net against a child that ignores stdin EOF. Keep it minimal but zombie-free.
