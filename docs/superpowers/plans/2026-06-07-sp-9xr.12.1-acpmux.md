# sp-9xr.12.1 — acpmux: in-container single-session ACP multiplexer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Relocate the ACP fanout from the node into the container as a standalone `acpmux` proxy so that MULTIPLE ACP clients share ONE `goose acp` conversation (the web ChatView via the node pump today; a terminal nori via `acpdial` in sp-9xr.12.2 next). `acpmux` is the single ACP **client** of `goose acp` (spawns it, owns one shared session) and presents canonical ACP as an **agent** on local TCP `:7000` (where `acpexec goose acp` listens today), multiplexing N downstream ACP clients onto that one upstream session — with replay of conversation history to late joiners, notification fanout, serialized prompts, and broadcast permission requests.

**Why canonical ACP downstream (not the node's Frame protocol):** the next task wires nori via a *trivial* `acpdial` raw-byte stdio↔TCP shim (nori's "agent" subprocess). For that shim to stay trivial, the socket it connects to must speak canonical ACP. The bonus: the node's EXISTING pump (`internal/node/pump.go`) — which is an ACP *client* upstream and a Frame *server* downstream to the web — sees `acpmux` transparently as "the agent" and works UNCHANGED. So this task changes only the image (dispatcher + new binary); **no node/CP Go changes**, and the existing `TestCPGooseAcpEndToEnd` must keep passing (now through acpmux).

**Architecture:**
- **Upstream (one stdio connection to `goose acp`):** ACP client. Port the proven patterns from `internal/node/pump.go`: `writeLoop`/`readLoop`, `call` (id+waiter), `sendPrompt` (in-flight id), busy/queue serialization, turn-end on the prompt result, and `session/request_permission` handling. Do `initialize` + `session/new` ONCE at startup (eager) → the shared session id `S`; cache goose's `initialize` result.
- **Downstream (N TCP clients, acpmux is the agent, canonical ACP):** an ACP *server*. Per client connection, handle: `initialize` → return the cached upstream initialize result; `session/new` (and `session/load`) → return `S` and **replay** the buffered `session/update` history to that client; `session/prompt` → enqueue onto the single upstream session (serialized); responses to `session/request_permission` → route the first answer back upstream. Fan every upstream `session/update` to all connected clients. Broadcast each upstream `session/request_permission` to all clients (first response wins; the rest are cancelled/ignored).
- **Replay log:** an append-only buffer of raw upstream `session/update` notification JSON with per-client cursors — the same shape as `pump.go`'s `log[]`/`base`/`seq`/`clientLoop`, but the stored items are raw ACP notifications and the wire is ACP, not Frame.

**Tech Stack:** Go (`internal/acpmux` core + `deploy/agent/acpmux` thin main), `internal/acp` for the wire (`acp.Message`, `acp.NewReader`, `acp.WriteMessage`), Docker image, real Docker + OpenRouter for the integration proof. **Run Go:** `export PATH="/var/home/alepar/sdk/go1.26.0/bin:$PATH"`.

### ⚠️ Design notes / risks
- **Eager upstream session.** acpmux spawns `goose acp` and does `initialize`+`session/new` at startup (before any client). The node (first client) then `session/new`s and gets `S` instantly. Keeps replay simple (the session always exists). goose idling from boot is fine.
- **Late-join replay in ACP terms.** ACP has no native "re-stream all prior notifications to a new client." acpmux provides history by replaying its buffered `session/update` log to a client right after it `session/new`s (or `session/load`s). For v1 this is a straight re-emit of the stored notifications; ordering preserved via the cursor.
- **session ids.** acpmux returns the REAL upstream `S` on `session/new` so downstream clients' `session/prompt` carry the id goose expects. acpmux can route on its single upstream session regardless.
- **Capabilities.** Return goose's cached `initialize` result to downstream clients so a client (later nori) sees goose's real `agentCapabilities`. Mapping `session/load` → (return `S` + replay) is a v1 approximation; the nori task (12.2) verifies it live and adjusts if nori needs more.
- **Prompt serialization across clients.** Two clients prompting → acpmux serializes via the SAME busy/queue logic as the pump (one in-flight `session/prompt`; queue the rest; drain on turn-end). v1 = serialized; no arbitration (one user across devices — see [[acp-shared-attach-architecture]]).

## Task 1: `internal/acpmux` core — upstream ACP client + downstream ACP-server multiplexer

**Files:** `internal/acpmux/acpmux.go`, `internal/acpmux/acpmux_test.go`

- [ ] **Step 1: Read the references.** Read `internal/acp` (the Message type + reader/writer) and `internal/node/pump.go` end-to-end. You are porting pump.go's upstream/serialization/permission logic and its replay-log/cursor mechanics, adapting the downstream wire to ACP.

- [ ] **Step 2: Define the core type.** `internal/acpmux/acpmux.go`, package `acpmux`. A `Mux` that:
  - Owns the upstream: `stdin io.Writer`, `stdout io.Reader` (the goose child's pipes), a `writeLoop`/`readLoop`, `call`, `sendPrompt`, `waiters`, `busy`/`queue`/`inflightPromptID`, `sessionID` (S), cached `initResult json.RawMessage` — ported from pump.go.
  - Owns the replay log of raw upstream `session/update` notification bytes: `log [][]byte` (or `[]logItem{seq, raw}`), `base`, `seq`, `maxLog`, and per-client cursors + `clientLoop` wake/replay — ported from pump.go's append/trim/cursor logic.
  - Owns downstream clients: `clients map[string]*dsClient` where a `dsClient` has its raw `net.Conn` (or an `acp` writer over it), a cursor, a notify channel, done channel — and a goroutine that replays buffered notifications + streams new ones.
  - Owns pending permissions: `pending map[string]*pendingPerm` keyed by the upstream goose request id, with the broadcast+first-wins logic ported from pump.go (`onPermissionRequest`/`resolvePermission`/`pickPermOption`), but forwarding to downstream as canonical ACP `session/request_permission` requests and routing the first downstream response back upstream.

- [ ] **Step 3: Upstream lifecycle.** `Start(ctx)`: launch `writeLoop`/`readLoop`; `initialize` (cache the result) + `session/new` → set `S`. `readLoop` dispatches: results to `waiters` (handshake) or, if it's the in-flight prompt id, `handleTurnEnd`; notifications → `session/update` appends to the replay log + wakes all clients; `session/request_permission` → `onPermissionRequest` (broadcast to downstream). Mirror pump.go's readLoop exactly, minus the Frame translation.

- [ ] **Step 4: Downstream ACP server.** `Serve(ln net.Listener)`: accept loop; per conn `go m.handleClient(conn)`. `handleClient`:
  - Register the client (assign an id), start its replay/stream goroutine (replays buffered `session/update` after the client establishes a session).
  - Read ACP messages from the conn (`acp.NewReader`): 
    - `initialize` → reply with the cached upstream init result (same id).
    - `session/new` / `session/load` → reply `{"sessionId": S}` (same id), then trigger the client's replay of buffered notifications.
    - `session/prompt` → extract the prompt text, run it through the shared busy/queue serializer (`fromClientPrompt`), reply to the client's prompt request id only at TURN END (mirror how goose replies to session/prompt after the turn — i.e., acpmux holds the client's prompt request id and answers it when the upstream turn for that prompt completes). NOTE: with serialization, a client's prompt may queue; answer its request id when ITS turn ends. Keep a small map prompt→clientReqID. (If exact per-client prompt-result correlation is hard for v1, it's acceptable to answer the prompt request at the next turn-end; document the simplification — the conversation content is delivered via fanned `session/update` regardless.)
    - response to a `session/request_permission` (the client answering acpmux's forwarded request) → `resolvePermission` (first wins → upstream).
    - `session/cancel` (notification) → best-effort forward upstream (optional for v1; may no-op).
  - On conn close → detach the client (close its goroutine), do NOT tear down the upstream (other clients/late reconnect continue).
  - **Zombie/leak discipline** (the sp-9xr.6/.16 lesson): every per-client goroutine exits on conn close; the upstream goose child is always reaped (`cmd.Wait()` in main); no goroutine leaks across connect/disconnect cycles.

- [ ] **Step 5: Unit tests with a FAKE upstream agent + multiple downstream clients.** `internal/acpmux/acpmux_test.go`. Build a fake upstream agent (an in-process goroutine wired to acpmux's upstream pipes via `io.Pipe`, speaking ACP): replies to `initialize` (caps), `session/new` (returns `"S1"`), on `session/prompt` emits a couple `session/update` `agent_message_chunk`s then replies to the prompt (turn end), and on demand emits a `session/request_permission`. Then connect TWO downstream ACP clients over an in-memory listener (`net.Pipe` or a real `net.Listen("tcp","127.0.0.1:0")`) and assert:
  1. **Fanout:** client A prompts; BOTH A and B receive the same `session/update` notifications.
  2. **Late-join replay:** client A prompts and gets updates; THEN client B connects + `session/new`s and receives the replayed history (the earlier `session/update`s) before/with new ones.
  3. **Serialized prompts:** A prompts and B prompts while A's turn is in flight; assert only one upstream `session/prompt` is in flight at a time and both turns complete in order.
  4. **Permission broadcast + first-wins:** the fake agent emits `session/request_permission`; BOTH clients receive a `session/request_permission`; the FIRST to respond resolves it (assert exactly one outcome reaches the fake agent; the second response no-ops).
  These are the load-bearing correctness tests — make them real, not tautological.
