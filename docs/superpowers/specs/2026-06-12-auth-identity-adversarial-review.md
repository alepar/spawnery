# Adversarial Review: Auth & Identity Design (2026-06-11)

**Reviewed:** `docs/superpowers/specs/2026-06-11-auth-identity-design.md` (GitHub Login, AS-Signed Sessions, Node-Verified Intents)
**Method:** multi-lens adversarial findings, each independently verified by a three-vote refutation panel against spec text, shipped code, prior roasts (web-epic 2026-06-11), and related specs (sp-ova node-auth, E4, owner-sealed-secrets). Severities below are panel-adjusted.
**Date:** 2026-06-12

## Verdict

The spec gets the architecture right — AS split from the CP, offline verification, PoP intents aimed at the untrusted-CP threat model — but its headline guarantee does not survive contact: the intent-signing mechanism is unimplementable and partially false as written (AC1), two of its recorded security claims are affirmatively wrong ("replay made inert by spawnId/generation binding"; "a stolen refresh cookie alone is useless"), and the whole lifecycle dimension (key rotation, revocation reach, refresh races, AS data loss, dev/prod parity) is missing. One critical and thirteen major findings survived the panel, most of them wire-freezing or claim-correcting, so they must land before A1/A4 freeze proto and token formats; three findings were downgraded to minor on verification and three were refuted outright. Nothing here argues for a different architecture — it argues the spec stops one design level above where its own threat model forces decisions to be made.

## Confirmed findings

### Critical

#### AC1 — Intent signatures cannot bind what the node executes; per-node jti permits cross-node and cross-restart replay

**Sections:** §5, A-D5, A-D7, §7 (slice A4); sp-ova §7a.
**Claim:** The compromised-CP-cannot-forge-workloads guarantee is unimplementable and partially void as specced. The four StartSpawn-causing intents have no defined signed byte tuple, no carriage of the raw signed bytes to the node, and no node-side correspondence check. Structurally the binding cannot hold: CreateSpawn's `spawn_id` is CP-minted in the response and generation is bumped server-side after signing, so the spec's "replay made inert by spawnId/generation binding" is false for CreateSpawn and half-false for Resume/Recreate/Migrate. The node executes a StartSpawn built entirely from CP-resolved values (image, app_ref, data_ref, generation) with no auth envelope, so a compromised CP holding one valid signed intent can substitute arbitrary parameters. Intents bind no target node, jti uniqueness is per-node (N-node fan-out of one signed intent; double-resume = journal split-brain), the jti cache has no restart-survival requirement, the freshness window has no number, and intent serialization re-opens the WM9 non-canonical Go↔TS trap the prior roast already amended for device-set entries (the token got the raw-bytes treatment; intents did not).
**Surviving evidence (3/3 panel votes confirm, all keep critical):** spec §5 lines 137–147; `proto/cp/v1/cp.proto:60-61` (spawn_id response-only), `:235` (Migrate target optional); `internal/cp/server.go:408-412` (uuid.NewV7 minted in handler), `:366-372` (CP resolves version→image); `internal/cp/lifecycle.go:276,304,451` (ClaimStarting bumps generation post-RPC); `proto/node/v1/node.proto:66` (StartSpawn has no token/signature/owner field); `internal/cp/scheduler/scheduler.go:60-65`; prior roast WM9 (web-epic review lines 67–72).
**Impact:** The threat the epic exists to close stays open via parameter substitution and multiplication of legitimately signed intents; A4 cannot be built without inventing the artifact format, carriage, and correspondence check — all wire-freezing decisions.
**Amendment:** Define a per-operation `SignedIntent` artifact with a domain-separation tag (`spawnery/intent/create-spawn/v1`), explicit field tuple (jti, issued_at, client-minted spawn_id or two-phase reserve-then-sign, image digest/app_ref pin, data_ref, mounts, explicit target node/class binding). The exact signed bytes travel verbatim inside StartSpawn; the node verifies the signature over those bytes and checks field-by-field equality against what it executes; CP-resolved fields are removed from node trust or echoed back for client signing. Pin the freshness window (60–120s); require the jti cache to cover it across restarts (persist, or refuse intents predating process start). Record any accepted residual (e.g. cross-node fan-out) explicitly.

### Major

#### AM1 — Migration resume-leg re-sign has no wire mechanism; as worded ("invisible") it invites a signing-oracle implementation

**Sections:** §5 (long operations), A-D5, A-D7, slice A4; web epic §3 W4/WM3.
**Filed critical; panel split 2–1 (one confirm-critical, one downgrade-major, one refuted). Recorded as major.**
**Claim:** §5 has the CP "request a fresh token + intent signature from the connected client for the resume leg (invisible via normal refresh)" — but MigrateSpawn is one synchronous, lock-held RPC (`cp.proto:235-236`; `lifecycle.go:361-414`), the only CP→browser channel is the raw-ACP WS relay with no signature-request frame (`ws.go:73-86`), and the wording conflates AS token refresh (client↔AS) with CP-solicited intent signing (client↔CP). No client-side correlation/validation rule appears anywhere in §5/A4/A5, so the naive implementation — sign whatever the CP sends, invisibly — makes the connected client a signing oracle for arbitrary StartSpawn-class intents under the spec's own CP-untrusted model.
**Surviving evidence:** quotes and code citations verified by all three panelists. The dissenting vote holds the secure implementation is the natural one (the web epic's migration flow is already client-driven and poll-shaped, and the client already drives the journal-key re-delivery "around" MigrateSpawn) — i.e. the gap is underspecification, not an inevitable exploit. That narrowing is why this lands at major.
**Impact:** A4/A5 must invent a protocol leg mid-implementation; the security-critical client behavior (never sign CP-supplied bytes) is unstated.
**Amendment:** Spec the leg client-driven: the initiating client pends the operation (spawnId, target, op kind, nonce), polls a pending-intent endpoint, constructs the resume-intent bytes itself from its pended record (CP-committed `{nodeId, generation}` echoed back and validated), signs single-use, submits. Clients MUST refuse and surface any solicitation not matching a locally pended operation. State whether MigrateSpawn becomes async or the CLI runs the poll-and-sign loop inside the blocking call; enumerate both legs' intents and verifiers.

#### AM2 — Silent /refresh rides a cross-origin cookie with no same-site constraint, no cookie attributes, no CSRF/credentialed-CORS spec

**Sections:** §2, §3, A-D3, A5; web epic D1/D2, WL5/WL6.
**Claim:** The web session hangs on an "HttpOnly refresh cookie scoped to the AS origin only" consumed cross-origin by the SPA, but the spec never states where the AS lives. Cross-site (any PSL host — including the staging-origin phase where A5 first runs), Safari ITP blocks and Firefox TCP partitions the cookie: login succeeds, silent refresh 401s forever — a browser-selective perpetual login loop. Even same-site, zero cookie attributes are specified (SameSite/Secure/Path; "origin-scoped" is not expressible by Set-Cookie), no Origin allowlist/CSRF posture on the one ambient-credential rotation endpoint (forged cross-site POSTs can interleave rotations and trip the reuse-kill — logout DoS + theft-log pollution), and no AS-side credentialed-CORS contract (exact-origin ACAO + ACAC:true) is assigned to any slice — A2's CORS item is CP-only; the WL6 amendment covered bearer RPCs, not cookies.
**Surviving evidence (3/3 confirm; panel downgraded critical→major):** spec lines 25, 96, 184 verified verbatim; greps confirm zero SameSite/CSRF/__Host-/registrable-domain content; web epic D1/D2 pins the SPA to a separate static origin. Platform facts (ITP/TCP, PSL suffixes, credentialed-CORS rules) verified. Downgrade rationale: the canonical deployment (auth.spawnery.dev next to app.spawnery.dev) is same-site, and the worst contingent security impact is DoS-grade.
**Impact:** Session persistence — A1/A5's central deliverable — is deployment- and browser-dependent with a silent total failure mode; A-D3's own rejection rationale (avoid credentialed CORS) is contradicted by the design's one remaining such endpoint, unowned.
**Amendment:** Mandate: AS and SPA share a registrable domain not on the PSL; cookie `Secure; HttpOnly; SameSite=Lax-or-Strict; Path=/refresh` (host-only, stated explicitly); strict Origin allowlist on /refresh; credentialed CORS for exactly the canonical SPA origin on /refresh + token endpoint. Add an AS-CORS item to A1; amend A-D3's row to acknowledge the residual; add a Playwright test for cross-origin /refresh including the cold-reload path.

#### AM3 — Strict reuse-kill revokes the family on honest races (multi-tab, lost responses, parallel spawnctl) — no grace window, no single-flight

**Sections:** §3 (/refresh step 2), §2, A3, A5.
**Claim:** "superseded_by set = reuse detected → revoke the entire family" has no idempotency or grace window. Honest conditions trip it deterministically: two tabs sharing the one cookie refreshing concurrently; a committed rotation whose response is lost (the client's only possible retry presents the superseded token — unfixable client-side); parallel spawnctl invocations against an unlocked shared state-dir file. Production rotation implementations ship reuse-leeway intervals for exactly these races. Post-outage, all clients refresh simultaneously into single-writer sqlite.
**Surviving evidence (3/3 confirm; panel downgraded critical→major — fail-closed, worst case is forced re-login):** spec lines 88–93, 50, 185–187 verified; greps for grace/leeway/single-flight/jitter/flock return nothing across all specs; no spawnctl locking precedent in code.
**Impact:** Sporadic "signed out everywhere" from routine use; scripts that self-destruct their credentials; false family-kills pollute the one theft signal the rotation scheme exists to produce.
**Amendment:** Bounded server-side grace: a /refresh presenting the most-recently-superseded token within ~30–60s returns the SAME cached successor pair (idempotent replay keyed on superseded_by); reuse outside the window or ≥2 generations old kills the family. Spec client single-flight (Web Locks/BroadcastChannel in the SPA; advisory file lock + atomic rename for spawnctl), refresh jitter, and WAL/queueing at the AS.

#### AM4 — No key id in the token + single-pubkey pinning everywhere makes AS session-key rotation fleet-breaking, compromise recovery unbounded

**Sections:** §3 (token format), §4, §5 (key distribution), A-D6; sp-ova OVA.2.
**Claim:** SessionTokenBody carries no kid/issuer; every verifier (CP + each node) pins exactly one AS session pubkey; "config-overridable for rotation" is the entire rotation story. No multi-key overlap window exists, so routine rotation has no instant at which the fleet can atomically swap pins — guaranteed partial interactive-auth outage, with self-hosted operators hand-editing config. Session-key compromise has no bounded recovery and no authenticated replacement channel (cannot be signed by the compromised key; role separation as written forecloses the PKI intermediate vouching). A-D6 rejected JWT for algorithm agility, but `kid` is key selection, not agility — discarded without replacement.
**Surviving evidence (3/3 confirm; panel downgraded critical→major — AS-trusted threat model makes the compromise leg an unrecorded residual, blast radius interactive-only, and kid is a backward-compatible proto addition):** spec lines 56–72, 28, 117, 158–160; `internal/sessiontoken/sessiontoken.go:45` (single-key Verify); node-auth spec has no session-key-compromise row.
**Impact:** The first real rotation — or the compromise response, when it matters most — is an unplanned fleet auth outage with no procedure.
**Amendment:** Add a short key-id to SessionTokenBody (alg stays hardcoded Ed25519); verifiers hold a small ordered SET of pinned pubkeys (current + next; trial-verify ≤2 as fallback). Spec routine rotation (pre-publish → overlap → switch → retire) and an emergency path: a replacement statement signed via the enrollment-pinned PKI chain (explicitly carving this one cross-role signature out of the separation rule), delivered through the CP but verified against the pinned root. Test rotation overlap hermetically.

#### AM5 — "A stolen refresh cookie alone is useless" is false: /refresh demands no proof-of-possession; minted tokens are plain bearer for the entire CP-local API

**Sections:** §3 (/refresh, threat shape), §5 (cnf paragraph), A-D3, A-D7.
**Claim:** No mechanism backs the §5 claim: /refresh is hash→row, reuse check, mint — no session-key signature, and the `refresh_sessions` schema stores no pubkey to check against (the binding is not just omitted but unimplementable as schemed). cnf is verified only by nodes on the five node-verified ops; per A-D7 every CP-local call is bearer. A cookie-only thief refreshes successfully, takes the rotated family, and holds full CP-plane control (Delete/Stop/Suspend/Rename/SetSpawnModel, Get/PutJournalKeyCiphertext, catalog mutation) renewably across the 30-day sliding window. "Theft-self-announcing" fires only when the *victim* next refreshes; attacker-first against an idle victim is silent, and the alarm then flags the victim's stale token.
**Surviving evidence (3/3 confirm, all keep major):** spec lines 88–94, 128–130; `proto/cp/v1/cp.proto:15-37`; `internal/cp/auth/auth.go:41-57`; `lifecycle.go:108/122/158` owner-bearer-gated only.
**Impact:** The recorded threat shape for the design's most sensitive credential is wrong; implementers will treat cookie theft as a non-event when it is silent CP-plane account takeover for up to 30 days.
**Amendment:** Require PoP at /refresh: fresh session-key signature over (refresh-token-hash, timestamp, nonce) verified against the family's registered pubkey (add the pubkey column) before rotation — making the claim true and killing CSRF-triggered rotation as a bonus. Otherwise strike the claim and record the residual honestly. Rewrite the §3 threat-shape paragraph either way.

#### AM6 — Web session key in evictable IndexedDB vs. a 30-day sliding cookie: guaranteed key-loss divergence with no detection, recovery, or cnf-key lifecycle

**Sections:** §3 (30d sliding window), §5 (client session keypair), A5; prior roast WM11.
**Claim:** The non-extractable P-256 key lives in IndexedDB; Safari ITP deletes script-writable storage after 7 days' non-interaction and Chrome evicts under pressure absent `navigator.storage.persist()` — while the HttpOnly cookie survives both. Cookie-alive/key-gone is unspecified and both branches are bad: with PoP-/refresh, silent unexplained logout; without, /refresh mints tokens whose cnf points at a nonexistent key — CP-local calls work, every node-verified op fails with opaque wrong-cnf refusals, presenting as flaky product bugs. No re-registration/rebind endpoint, no in-family key rotation, no absolute family max age (a quietly exfiltrated CLI state dir slides forever), no key-presence self-check, no distinct cnf-mismatch error code. The prior roast established this exact substrate failure (WM11) and the web epic was amended — for device keys only; this spec puts a second critical key into the same substrate with zero carried-over mitigation.
**Surviving evidence (3/3 confirm major):** spec lines 84, 125–130, 187; web-epic amendment at its lines 126–129 covers device keys only; platform facts verified (cohort is >7-day-gap Safari returners, not literally all weekly users).
**Impact:** Recurring whole-cohort half-authenticated breakage (logged in, can delete spawns, cannot start or open any) that A5 hasn't designed for.
**Amendment:** In §5/A5: request persistent storage at key creation; on every session restore, positively verify the key can sign (test signature) BEFORE /refresh; key missing ⇒ revoke family server-side, route to clean re-login (cheap by design — say so); document the Safari ≥7-day-absence re-login as accepted. Add an absolute family max age (e.g. 90d), optional in-family key rotation (new pubkey signed by old key), and a distinct cnf-mismatch error code so clients drive recovery.

#### AM7 — The headless flow is not RFC 8628 — it is an out-of-band bearer-code handoff, a phishable antipattern, with no cnf-registration home

**Sections:** §2, §7 (slice A3).
**Claim:** RFC 8628 runs the other way: the device initiates, the USER enters the low-entropy code AT the AS, the device polls. The spec's flow (AS device page mints a code the user pastes INTO spawnctl) is the deprecated OOB pattern Google killed: an AS-minted, transferable, account-scoped bearer credential with no specified entropy, TTL, rate limit, or binding to a pending spawnctl request. "Paste me the code from your device page" is a one-message account takeover that also bypasses the cnf layer at issuance (the attacker's spawnctl registers its own session key). The §5-required session-pubkey registration has no defined place in this flow at all — headless-minted tokens as specced could not drive any node-verified op.
**Surviving evidence (3/3 confirm major):** the entire headless spec is one clause in §2 plus "headless paste" in A3; §5 registration is loopback-flow-only; RFC 8628 §3.1–3.5/§5.4 direction confirmed. Panel caveat: real-world 8628 is also phished — the AS-side authenticated account-confirmation step carries the protective weight, not direction alone.
**Impact:** A3's headless leg ships either a hand-wave or a phishable bearer-code grant on the path the epic exists to harden.
**Amendment:** Spec the real device grant: spawnctl POSTs its session pubkey to an AS device-authorization endpoint, receives `{device_code, user_code, verification_uri}`; user confirms at the AS in an authenticated browser with account context displayed; spawnctl polls the token endpoint; user_code short-TTL and rate-limited.

#### AM8 — No state / login-CSRF / redirect_uri-validation story on either OAuth leg

**Sections:** §2, §7 (A1/A3/A5).
**Claim:** Two chained code flows, zero mention of `state`, callback transaction binding, or redirect_uri validation. (1) Login CSRF/session fixation on the AS↔GitHub leg: a forged callback logs the victim silently into an attacker-owned account — and PKCE on the client↔AS leg does not block code injection into the victim's own in-flight AS authorization, so subsequent device-set registrations, sealed secrets, and spawns land where the attacker reads them. (2) The SPA-side state (or explicit PKCE-as-CSRF) check is unspecced. (3) spawnctl's `127.0.0.1:<random>` requires the RFC 8252 §7.3 loopback variable-port allowance and ONLY that relaxation; nothing says so. §7's tests omit forged-callback/code-injection negatives.
**Surviving evidence (3/3 confirm major):** the OAuth `state` parameter appears nowhere in the document ("state" occurs only as "state dir"); E4 and sp-ova contain nothing; not covered by any prior roast.
**Impact:** Session fixation defeats the epic's purpose — every accountId-keyed artifact binds to an attacker-controlled account.
**Amendment:** Add to §2/A1: per-request state (or PKCE-as-CSRF on the client↔AS leg with an explicit SPA-side check), AS-side browser-session-bound state on the GitHub callback, exact-match redirect_uri registration with the sole RFC 8252 loopback-port exception. Add forged-callback and code-injection negatives to §7.

#### AM9 — githubSub is undefined; keying on GitHub `login` is a recyclable-username account-takeover footgun

**Sections:** §1, §2, §7 (A1); E4 §2.
**Claim (as it survived):** GitHub user OAuth is plain OAuth 2.0 — no ID token, no `sub` — so `githubSub` (used once, never defined) and E4's "linking accounts is by OAuth subject" name a field that does not exist on GitHub's wire. The only GitHub identity material the spec mentions is the mutable, re-registrable `login`: release a username, an attacker registers it, logs in, and inherits the victim's accountId, spawns, device-set chain, and sealed-secret registry. The GitHub-App user-to-server 8h token expiry needs recording against the re-asserted E4 dual-duty claim.
**Struck by panel:** the original "GitHub ignores PKCE" sub-claim is stale — two panelists verified GitHub shipped validated PKCE (S256) for OAuth Apps and GitHub Apps on 2025-07-14, so the proposed "no PKCE — GitHub ignores it" rewrite must NOT be adopted. GitHub still requires `client_secret` even with PKCE, which is worth one clarifying clause.
**Surviving evidence (3/3 confirm; severity votes 2 major / 1 minor → major):** spec §1 line 34; E4 line 38; GitHub rename/recycling behavior confirmed.
**Impact:** Undefined root identity key resolvable the wrong way into silent account takeover.
**Amendment:** Define `githubSub := GitHub numeric user id` (int64/digit-string), never `login`; clarify the AS↔GitHub leg is a confidential client (client_secret + state; PKCE optional-supported); record the user-to-server token-expiry caveat against the E4 dual-duty claim.

#### AM10 — Logout and family revocation sever nothing live: open WS sessions survive "sign out everywhere" indefinitely

**Sections:** §3, §4, §5, A-D5.
**Claim:** Revocation touches one AS table; CP and nodes verify offline against pinned pubkeys with no revocation channel, so a revoked family's last token works everywhere for its full 15 minutes — a consciously recorded tradeoff — but the unrecorded part is worse: "open sessions survive (token authorizes session open, SSH-style)" means an attacker's already-attached terminal into the victim's spawn persists with no bound after family-kill, which is the one thing the user invoking stolen-laptop response is trying to achieve. `ws.go` checks the token once at bind, then relays raw bytes forever. Web logout also cannot clear the HttpOnly cookie; no /logout endpoint exists in any slice.
**Surviving evidence (3/3 confirm major):** spec lines 95, 104–105, 117–118; `internal/cp/ws.go:42` (single bind-time check). Panel notes the ≤15-min token tail is recorded and the dead-cookie point is hygiene-grade; the live-session gap alone carries the severity, exactly the prior roast's WM7 class (false assurance in the incident-response flow).
**Impact:** The headline theft response is cosmetic against the highest-value asset — a live session into a spawn.
**Amendment:** Spec a CP-side hook: family/account revocation propagates to the CP (AS→CP signed revocation event or short-poll); CP terminates WS sessions and in-flight relays bound to revoked token_ids; cap session lifetime with periodic in-band token re-presentation (~15 min, aligned with refresh). Define AS /logout (revoke + Set-Cookie expiry).

#### AM11 — cnf/intent wire facts unpinned: signature encoding, SPKI hashing, pubkey carriage, spawnctl algorithm, SessionOpen proto gaps

**Sections:** §5, §7 (testing), A-D6, slices A4/A5; prior roast WM9.
**Claim:** On the new Go-verifies-WebCrypto path: (1) WebCrypto ECDSA emits raw P1363 r||s, Go verifies DER (`SignASN1` is the shipped seal-code choice) — WM9's amendment mandated the conversion sentence for device keys; this spec opens a parallel path and drops it. (2) cnf is "SHA-256 of the session pubkey SPKI" but the codebase's only P-256 pubkey precedent is SEC1-uncompressed — wrong-bytes-hashed risk on the Go side. (3) The cnf is a hash, so nodes need the FULL pubkey to verify intent signatures — no message in §5/A4 carries it; node verification is unimplementable as written. (4) spawnctl's session-key algorithm is unstated; an Ed25519 pick (the natural Go default, given A-D6's "one algorithm" language) forces verifier algorithm dispatch. (5) The signed SessionOpen tuple needs `generation` but no client-facing surface exposes it (SpawnSummary lacks it; router builds SessionOpen without it; for CreateSpawn the generation doesn't exist at signing time), omits `session_id` (compromised CP can rebind one signed open within the spawn), and node.proto SessionOpen/StartSpawn have no token/signature fields — proto changes A4 never enumerates.
**Surviving evidence (votes 2 major / 1 minor → major):** spec lines 69, 126–131, 137, 143–145; `seal/device.go:18-21, 214-217`; `deviceset.go:95`; `cp.proto:224`, `router/router.go:93`, `node.proto:66/70`. Panel caveats: §7 does mandate WM9-style Go↔TS vectors, and DER SPKI is deterministic — items (3)–(5) are the load-bearing gaps.
**Impact:** A4/A5 freeze wire format; each item is a cross-language interop break or a verifier-agility regression the project already paid to learn once.
**Amendment:** Pin: session keys are ECDSA P-256 for ALL clients including spawnctl; one signature encoding stated in spec text (P1363 with conversion at the Go boundary, or DER at the WebCrypto boundary); cnf = SHA-256 over DER SPKI exactly as `exportKey('spki')`/`x509.MarshalPKIXPublicKey` emit, with vectors; the full SPKI travels in the intent/SessionOpen envelope, node checks hash-match before verifying. Enumerate the proto additions (auth envelope on StartSpawn/SessionOpen, generation exposed client-side, session_id in the signed open). State the system has exactly two fixed algorithms, one per role (Ed25519 = AS signing; P-256 = client cnf).

#### AM12 — Dev mode runs a structurally different auth pipeline; CP-side signature-threading bugs of the sp-gzvo class stay invisible until enforced e2e

**Sections:** §2 (dev mode), §4, §5 (modes), §7.
**Claim:** In `just dev`: raw CP_DEV_TOKENS (no SessionTokenBody, no cnf), no AS, no keypair, no intent signatures, `NODE_AUTH_MODE=insecure` *skips*. The entire A4/A5 data path is dead code in the only stack developers run daily, and dev tokens structurally cannot carry a cnf — forcing a sign/don't-sign SPA fork. The CP threading seam (lifecycle→`scheduler.Provision`→StartSpawn) is precisely where sp-gzvo shipped an unthreaded-field bug two days before this spec, past unit tests that hand-built the field; the same omission for token+signature passes `just test`, `just dev`, and Playwright (un-enforced spawnlet), failing only on the hosted enforced fleet — as the opaque "spawn failed to start" of AM13's sibling finding.
**Surviving evidence (3/3 confirm major):** spec lines 51–52, 118, 162; commit 6eab9df (sp-gzvo, same Provision seam, with in-code warning comment); §7's only enforced coverage is a host-gated e2e negative that passes with broken CP threading.
**Impact:** A prod-only breakage class for the epic's central mechanism, with zero dev or hermetic reproduction.
**Amendment:** Close the divergence: dev clients ALWAYS mint the session keypair and sign intents (CP-boot-generated dev AS keypair or the fake provider mints cnf-bearing tokens in dev); redefine `insecure` as verify-and-log-don't-enforce so missing/garbage signatures are loud in `just dev` logs; add a hermetic CP test asserting all four lifecycle handlers thread token+signature into the node-bound StartSpawn.

#### AM13 — AS data-loss story absent: losing one sqlite file permanently orphans every accountId-keyed artifact; signing-key custody unspecified

**Sections:** §1, §3 (AS-offline), §6 (AS persistence).
**Claim:** The spec details AS-*offline* behavior and says nothing about AS *data loss*. The githubSub→accountId mapping exists only in one sqlite file while accountId is the universal key for CP owner_id, node-cert SANs, enrollment tokens, device-set chains, and sealed-secret AADs. Lose the users table and re-login mints a NEW UUID: ownership of everything orphans with no re-binding mechanism specified — not a 15-minute fuse. Lose the refresh table = global logout AND all device-set chains (same DB); the WM6 monotonic chain-head pin means even a stale-backup restore hard-fails pinned clients, so restore *semantics* need design, not just backups. "Smallest, cheapest-to-make-redundant component" is asserted, never designed; session-signing-key custody/escrow is unspecified (shipped default is an in-memory generated key). The prior CP-backup mandate (sp-jf7 lineage) covered exactly this tier-0 data and was silently dropped when the AS split relocated it.
**Surviving evidence (3/3 confirm major):** §6 line 176–177, §1, line 109; `seal.go:72-80` AAD binding; greps confirm zero backup/RPO/restore/escrow content anywhere. Panel caveats: AAD binding severs the authz path, not cryptographic decryptability; manual operator re-bind is conceivable — which is exactly the decision the spec must record.
**Impact:** A single-file loss event converts to platform-wide ownership destruction; key loss compounds it through the unspecced rotation of AM4.
**Amendment:** Add an AS durability section: backup cadence + restore drill (litestream/snapshot, stated RPO) with users + device-set tables as tier-0; signing-key custody (perms, offline escrow, relationship to the PKI ceremony); a recovery matrix for {DB lost, key lost, both}; an explicit decision on whether re-binding a fresh accountId to surviving artifacts is supported recovery or a non-goal; restore-vs-chain-head-pin semantics.

### Confirmed, panel-downgraded to minor

#### MC1 — SessionTokenBody collides with shipped `internal/sessiontoken`: same AS key role, identical b64url.b64url wire shape, no domain separation in either

**Sections:** §3, A-D6; sp-ova §7a. (3/3 confirm; all downgrade major→minor — the legacy artifact has zero production verifiers and a dual-grammar forgery is practically unreachable, but the unrecorded supersession plus missing domain-separation discipline the repo itself follows elsewhere, plus unpinned base64url padding, are real spec-hygiene defects.)
**Amendment:** State explicitly that `internal/sessiontoken` is superseded and deleted by A4, or give every Ed25519-signed AS artifact a mandatory domain-separation prefix over the signed bytes plus a type/version field; pin base64url as unpadded RawURLEncoding.

#### MC2 — Single audience "api" for CP and nodes: every token relayed node-ward is a full-power CP-API bearer credential in the node's hands

**Sections:** §3 (audience), A-D7, §5; sp-ova §9. (Contested: votes confirm-minor / refuted / confirm-major. Recorded as confirmed-minor.)
