## 1. Executive summary

**Yes — we can give the agent an unrestricted, namespaced-root writable rootfs whose delta survives same-node resume and migrates via Kopia, across both lanes — but the win is gated on policy (idmap sizing, ownership normalization, quota) far more than on capabilities.** User namespaces are the correct mechanism to make `apt`/`dpkg`/`useradd`/`chown -R`/`chmod` "just work": a userns init starts with the *full* namespaced capability set by construction, so no custom cap list is needed — the binding constraint is a **populated multi-uid subid range** (≈65536-wide) plus a **permitted `setgroups`**, not "add a capability" (`--cap-add CAP_SYS_ADMIN` provably does *not* fix `apt`). The recommended capture mechanism is **delta-as-OCI-layer**: `docker commit` on the Docker lane and a containerd snapshot **diff** on the CRI lane, both normalizing to one portable OCI image layout that ships delta-only through the existing Kopia→Garage journal and resumes as `pinned-base + delta` at pod recreate. This mechanism has direct, shipped prior art (Modal filesystem snapshots are "diffs from your base image" that are themselves reusable images), and it deliberately avoids the E2B/Firecracker memory-snapshot tax (~4 s/GiB RAM, cross-kernel-restore fragility) while closing the Codespaces data-only gap (apt/chown/rootfs edits lost on rebuild). The Docker/runc lane is on **solid primary-sourced footing**; the **CRI/containerd diff mechanics, the runsc `--overlay2` host-visibility question, multi-GB commit latency, userns uid-shift normalization, and cross-lane tar interchange are not yet sourced** and are the gating unknowns. Same-node suspend should additionally keep a **zero-cost "stop-but-keep"** local path (engine retains the writable layer in place) and never touch the journal; **raw upperdir harvesting is rejected as a migration mechanism** (it re-introduces a `CAP_SYS_ADMIN`-on-restore requirement that cap-dropped restorers provably cannot satisfy). Finally, userns is **necessary-but-not-sufficient** security for the untrusted multi-tenant lane — that lane must run under `runsc` (Sentry-mediated syscalls), which is the strongest, best-sourced result in the pass.

## 4. Section writeups

### S1. User namespaces per engine — the support matrix

This section validates the *mechanics* of decision #2 (give the agent namespaced root via user namespaces so `apt`/`dpkg`/`useradd`/`chown -R`/`chmod` "just work") against Spawnery's two lanes (Docker/runc today, containerd/CRI+`runsc` on cloud nodes), the pinned-base + delta-capture model, and the world-writable-0777 mount workaround it is meant to replace. The headline finding: **user namespaces are the correct mechanism, but only with an adequately *sized* subordinate id range and a *permitted* `setgroups` — a single-uid map or a "just add the capability" approach provably does not work.**

**The capability red herring: why `--cap-add` alone cannot fix apt.** The most load-bearing negative result for Spawnery's design is that **adding capabilities — even `CAP_SYS_ADMIN` — does not make `apt-get update` work in a rootless/userns container.** The runc issue [#2517](https://github.com/opencontainers/runc/issues/2517) is literally titled "Cannot run apt-get update in rootless container with sufficient capabilities." The failure is not a missing capability; it is a uid-mapping failure. `apt` deliberately drops privilege to its sandbox user `_apt` (uid 100) for download methods via `setgroups`/`setegid`/`seteuid`, and under a too-narrow userns those calls fail:

```
setgroups 65534 failed - setgroups (1: Operation not permitted)
setegid 65534 failed - setegid (22: Invalid argument)
seteuid 100 failed - seteuid (22: Invalid argument)
```

The `EINVAL` (errno 22) on `setegid`/`seteuid` means the target uid 100 (`_apt`) and gid 65534 (`nogroup`/`nobody`) are **not mapped** into the namespace — per `seteuid(2)`/`user_namespaces(7)`, an id that is not valid in the userns yields `EINVAL`. The reporter's failing config was a single-uid map (`"containerID": 0, "hostID": 1000, "size": 1`), so only container-root existed inside the namespace and every other uid `apt` needed was unmapped ([#2517](https://github.com/opencontainers/runc/issues/2517)).

**Implication for Spawnery:** the spawn's userns must allocate a *multi-uid subordinate range*, conventionally a full 65536-wide range (e.g. `100000:65536`), wide enough to cover both uid 100 (`_apt`) and gid 65534 (`nobody`). Note the classic off-by-one: covering uid 65534 requires the range to extend to it (Podman's `--userns=auto:size=65535`/full 65536 idioms exist for exactly this). The corollary stated positively: **"userns + a sufficient subid range makes apt work" is TRUE; "add a cap and apt just works" is FALSE.** An escape hatch that sidesteps the issue entirely is setting `APT::Sandbox::User "root"`, which disables apt's privilege-drop so no `_apt` switch is attempted — worth keeping in the spawn's base-image config as a belt-and-suspenders measure ([#2517](https://github.com/opencontainers/runc/issues/2517)).

**The `setgroups` deny trap — a containerd-version-specific footgun.** Beyond mapping the right *range*, the namespace must also have **`setgroups` permitted**. The `setgroups 65534 failed - Operation not permitted` (`EPERM`) line above is governed separately from the id map: it is the kernel's userns `setgroups`-deny policy, controlled by `/proc/{pid}/setgroups`. Whoever writes the gid_map must permit `setgroups` for apt's group-drop to succeed. This is a live, version-specific footgun in the CRI lane. containerd [#10742](https://github.com/containerd/containerd/issues/10742) documents that PR #10611 switched gid_map population from custom `/proc/pid/gid_map` writes to Go's stdlib `forkAndExecInChild`, which writes `'deny'` to `/proc/{pid}/setgroups` **unless `SysProcAttr.GidMappingsEnableSetgroups` is `true`** (stable Go behavior since 1.9). Before #10611, `setgroups` was left at the default `allow`; the regression made it `deny` ([#10742](https://github.com/containerd/containerd/issues/10742); fix tracked as PR #10741). **Caveat / contested:** the original sharper claim — that "in recent containerd 2.x the constructed userns *fails* `setgroups` with `EPERM`" — did not survive verification and should NOT be asserted as a standing fact. Treat it as: a *known regression existed* on the containerd main/2.x line tied to #10611, with a fix in flight (#10741). Spawnery's CRI integration must pin a containerd build where this is fixed, or verify `setgroups`=`allow` in the constructed pod userns as part of node bring-up.

**Docker lane: userns is daemon-wide, not per-spawn.** A structural constraint for the Docker/runc lane: **Docker's user-namespace remapping is daemon-wide, not per-container.** Enabling it via `dockerd --userns-remap` or `daemon.json` `{"userns-remap": ...}` makes *all* containers start with user namespaces by default; the only per-container knob is opt-*out* via `--userns=host`. There is no per-container syntax to supply a custom subuid/subgid map ([docs.docker.com/engine/security/userns-remap](https://docs.docker.com/engine/security/userns-remap/): "If you enable user namespaces on the daemon, all containers are started with user namespaces enabled by default"). Consequences for Spawnery: the remap is a host-level posture applying uniformly to agent *and* sidecar; mapping driven by `/etc/subuid`/`/etc/subgid` entries `user:beginUID:count` (e.g. `dockremap:231072:65536` maps container root to host uid 231072). This single daemon-wide range is conveniently exactly the wide range the apt fix requires — so the Docker lane gets the right range "for free" once remap is on, but loses per-spawn id-range isolation. Because the range is ≥65536, uid 100 and gid 65534 fall inside it and apt's privilege-drop works — provided `setgroups` is permitted (Docker/runc handle this correctly, unlike the containerd regression). If per-spawn id isolation is desired, **rootless Podman** is cleaner: the launching user maps to uid 0 by default ([docs.podman.io run.1](https://docs.podman.io/en/latest/markdown/podman-run.1.html)), and `--userns=auto` allocates a distinct subid range per container — aligning with "rootless Podman desired later."

**Bind mounts under userns: the fate of the 0777 workaround.** Under userns-remap, **the agent does NOT see uid-0 ownership on host bind mounts** unless host files are arranged into the mapped subuid range. Docker's docs flag this: "if volumes are mounted from the host, file ownership must be pre-arranged if you need read or write access to the volume contents" ([docs.docker.com userns-remap](https://docs.docker.com/engine/security/userns-remap/), known limitations). With map base 100000, a `/app` file owned by host uid 0 falls *outside* `[100000, 165536)` and appears inside as the overflow uid (65534/`nobody`). Three resolutions: (1) **chown-into-range** on the host before mount — cleanest semantics, but spawnlet must know the map base and re-chown on Finalize; heavier with per-spawn ranges; (2) **idmapped bind mounts** — kernel applies a uid/gid shift at the mount, so host-uid-0 files appear as uid 0 in-range without rewriting on-disk ownership; strictly better long-term (no destructive chown, composes with per-spawn ranges) but carries kernel+fs requirements; (3) **world-writable 0777** (current workaround) — *survives* but degrades: 0777 is writable by any mapped uid so it keeps working, but it is the weakest (no ownership, weak isolation). **Nuance:** plain *read* access needs only the world-readable bit; the in-range requirement bites only for *write/owner-mode* access — so read-only `/app` mounts need no special handling. **Secrets tmpfs:** same mapping applies — mount with ownership in the spawn's subuid range (or idmapped); "must never persist" is orthogonal to userns and unaffected.

**The security bound — necessary-but-not-sufficient for multi-tenant.** Namespaced root is genuinely bounded: an escapee "is running as an unprivileged high-number UID on the host, which does not even map to a real user" ([docs.docker.com userns-remap](https://docs.docker.com/engine/security/userns-remap/)); "containers running in a user namespace cannot have more privileges than the user that launched them" ([docs.podman.io](https://docs.podman.io/en/latest/markdown/podman-run.1.html)). The kernel model (`user_namespaces(7)`) confirms a userns capability permits privileged ops *only* on namespace-governed resources. **Two precision caveats:** (a) "does not map to a real user" is a property of *correct configuration* (dedicated unprivileged subid range), not automatic; (b) the userns capability bound is the *intended* model, not bulletproof — unprivileged userns is a documented LPE attack-surface amplifier: CVE-2023-0386 (OverlayFS copy-up), CVE-2022-0492 (cgroup `release_agent`), GameOver(lay) all reached real host root. **This motivates the lane split:** userns alone is acceptable for the trusted-ish single-tenant Docker lane, but the untrusted multi-tenant cloud lane must run under **gVisor/`runsc`** *in addition to* userns.

**Bottom line.** *Docker lane (today):* daemon-wide `userns-remap` with ≥65536 range makes apt/dpkg/useradd/chown work, replacing cap-drop-ALL failures; cost is daemon-wide (no per-spawn isolation) and writable mounts + secrets tmpfs need chown-into-range or idmapped mounts (0777 still functions, weakest). *CRI/containerd lane:* userns via KEP-127, but verify past the #10611 `setgroups`-deny regression and pair with `runsc`; idmapped mounts preferred for `/app`, subject to kernel/fs prereqs. *Both lanes / delta capture:* none of this changes the portable-artifact contract — userns only changes the *ownership of files inside* the delta (owned by mapped uids); the spawnlet must round-trip ownership consistently (idmapped mounts cleanest) so a delta captured on node A with base X restores on node B with a different base.

### S2. Delta capture mechanics

The design captures the agent container's writable-layer delta as an OCI layer (`docker commit` on Docker/runc; a containerd snapshot **diff** on CRI), ships only that delta through the Kopia→Garage journal on migration, and resumes as pinned-base + delta. The Docker-lane mechanics are well-sourced and sound; the CRI/runsc-lane mechanics and large-delta latency are **not yet sourced**.

**Docker lane: `docker commit` semantics — quiescing.** By default, `docker commit` pauses the container and all processes for the commit's duration, explicitly "to reduce the likelihood of encountering data corruption" ([docs.docker.com commit](https://docs.docker.com/reference/cli/docker/container/commit/)). This default is exactly what Spawnery wants for a consistent suspend snapshot. The pause can be disabled — current flag `--no-pause`; the legacy `-p, --pause` (default true) still works but is **deprecated as of Docker CLI v29.0** in favor of `--no-pause`, removal targeted v30, passing both is an error. **Recommendation:** keep the default pause; the freeze window is a feature for suspend-and-snapshot, and committing a live container re-introduces the corruption risk. Because pods are recreated per generation, the brief freeze is invisible.

**What is and is not captured.** `docker commit` snapshots only the writable filesystem layer plus config; it does **not** include "any data contained in mounted volumes" ([docs.docker.com commit](https://docs.docker.com/reference/cli/docker/container/commit/)). This exclusion applies to **both** named volumes and host bind mounts — precisely how Spawnery's storage model is built. Data mounts (`/app/<path>`) and the secrets tmpfs live outside the writable layer, so they are **not** captured. This is load-bearing: the delta is disjoint from journaled data mounts (no double-capture) and the secrets tmpfs cannot leak into a persisted/migrated layer. Verify in tests, but the Engine storage model guarantees it.

**Latency on multi-GB deltas.** *Not sourced.* No verified measurement of `docker commit` wall-clock on multi-GB writable layers. Since the default pause holds the agent frozen for the commit's full duration, commit time *is* the quiesce-window length and must be benchmarked against realistic agent deltas (build caches, node_modules) before committing to same-node suspend SLAs.

**Repeated suspend: layer-chain growth and squash threshold.** Each suspend/commit adds one lower layer. Docker `overlay2` "natively supports up to 128 lower OverlayFS layers," tied to commit performance ([docs.docker.com overlayfs](https://docs.docker.com/engine/storage/drivers/overlayfs-driver/)). That 128 is the **driver's** documented max, not a kernel constant — the canonical kernel doc states **no numeric maximum** ([kernel.org overlayfs.txt](https://www.kernel.org/doc/Documentation/filesystems/overlayfs.txt)). The real limit is the ~4 KB mount-option buffer holding the concatenated `lowerdir=` list, which empirically fails *below* 128 (observed near 122). **Treat the squash threshold as ~120 stacked deltas, not 128**, and squash before approaching it. This bound is specific to the overlay2 lane; the containerd overlayfs snapshotter shares the kernel stacking constraint, while gVisor/runsc does not use host OverlayFS the same way and must be validated separately. In practice every migration is a squash opportunity: resume on node B collapses pinned-base + delta(s) into a new pinned base, so chains grow only under repeated *same-node* suspend/resume.

**Whiteout, opaque-directory, and xattr fidelity.** A captured delta must faithfully encode deletions or deleted files reappear on resume. *copy_up is the delta* — first write copies a lower file into the upperdir, which is the writable-layer delta ([kernel.org overlayfs](https://docs.kernel.org/filesystems/overlayfs.html)). *File deletions → whiteouts* — a char device with **0/0 device number**, or a zero-size regular file with `trusted.overlay.whiteout` xattr ([kernel.org](https://docs.kernel.org/filesystems/overlayfs.html), [overlayfs.txt](https://www.kernel.org/doc/Documentation/filesystems/overlayfs.txt)). *Directory deletion → opaque dirs* — `trusted.overlay.opaque='y'`; drop it and the base's deleted dir contents wrongly re-merge ([overlayfs.txt](https://www.kernel.org/doc/Documentation/filesystems/overlayfs.txt)). **Key precision:** these on-disk overlayfs encodings are **not** what ships in the OCI layer. `docker commit` (and the containerd diff path) serialize the upper into **OCI image-spec tar conventions**: `.wh.<filename>` to mask a deletion, `.wh..wh..opq` inside a directory whose lower contents must hide ([OCI layer.md](https://github.com/opencontainers/image-spec/blob/main/layer.md)). So capture is a **format translation**, not a byte-for-byte tar of the upperdir; resume reverses it on unpack. Both Docker's and containerd's archive/diff code already do this — implementers must use those code paths and **not** hand-roll a raw upperdir tar (which loses deletions). Round-trip tests should assert a deleted file and a replaced (opaque) directory survive suspend→migrate→resume.

**CRI / containerd lane: snapshot diff.** *Largely unsourced — flagged open work.* The intended mechanism (produce a diff layer from the CRI container's snapshot via the containerd Go diff service in the `k8s.io` namespace, discovering the snapshotter key from the container id; stopped-but-not-removed vs removed; required privileges) is **not** covered by any verified source. The one anchored fact: containerd's `archive`/`diff` package performs the same overlayfs↔OCI whiteout/opaque translation, so the *artifact* converges (lane-specific capture code is acceptable per decision #4; lane-specific artifacts are not). Everything else — namespace interop, key discovery, lifecycle/privilege — must be validated against the containerd API before it can be specced.

**runsc (gVisor): does the host snapshotter ever see the writes?** *Unsourced — flagged open work.* With `runsc --overlay2` enabling an in-sandbox rootfs overlay (recent versions default toward a `root:self`-style medium), it is an **open empirical question** whether agent writes ever land in the host containerd snapshotter's upperdir — i.e. whether a host-side `containerd diff` can observe any delta, or whether the in-sandbox overlay must be disabled/host-visible for capture, and at what gofer-I/O cost. Must be tested on a runsc node (write in-agent, then attempt a host snapshotter diff) before the CRI/gVisor capture path is viable.

**Capture timing and junk exclusion.** The default `docker commit` pause gives a clean quiesce point; per-generation-recreate means no need to capture a *dying* container. **Junk exclusion** (`/var/cache/apt`, build caches) at capture time is **not** addressed by any verified source. Because userns makes in-container apt/chown/chmod work, agents will routinely populate large worthless caches that bloat every delta and migration. A capture-time exclusion strategy should be designed — pre-commit cleanup hook, `.dockerignore`-equivalent at the diff layer, or mounting cache dirs as excluded tmpfs/volumes so they fall outside the writable layer (the cleanest, since mount-exclusion is already proven for data mounts and secrets tmpfs).

**Bottom line.** The **Docker/runc lane is on solid, primary-sourced ground**: pause-by-default gives a consistent snapshot, mounts and secrets tmpfs are provably excluded, the overlay chain is bounded (~120 practical, squash before that), whiteout/opaque fidelity is handled by `docker commit`'s OCI translation. The **CRI/containerd diff mechanics, the runsc `--overlay2` host-visibility question, multi-GB commit latency, and junk exclusion are not yet sourced** and gate this section.

### S3. Resume + cross-node replay

This covers reassembling pinned-base + delta on a target node, cross-lane portability (Docker capture → containerd apply and vice versa), how the artifact rides Kopia→Garage, and how each lane keeps the pinned base digest alive against GC. OCI *packaging and apply* are on firm primary-spec footing; *capture-side normalization*, *userns uid-shift*, *Kopia dedup*, and *lease-based pinning* are materially less verified.

**Artifact shape: an OCI image layout, not a bare layer tar.** The portable form is an **OCI image layout** — three required components: a `blobs/` directory of content-addressable blobs, an `oci-layout` marker carrying `imageLayoutVersion`, and an `index.json` entry point ([image-layout.md](https://github.com/opencontainers/image-spec/blob/main/image-layout.md)). Each blob lives at `blobs/<alg>/<encoded>` where `<encoded>` is the content digest; "The content of `blobs/<alg>/<encoded>` MUST match the digest." A common trap: a complete artifact is **not** just "write the delta tar into `blobs/` and point `index.json` at it." You must also synthesize an updated image **config** blob (`rootfs.diff_ids`/`history` extended) and a **manifest** blob referencing `config + layers`, with `index.json` pointing at the manifest. Layers are ordered: "The array MUST have the base layer at index 0. Subsequent layers MUST then follow in stack order" ([manifest.md](https://github.com/opencontainers/image-spec/blob/main/manifest.md)). So the pinned base sits at index 0 and captured deltas append in capture order. The delta tar must use a registered media type — `application/vnd.oci.image.layer.v1.tar`/`.tar+gzip`/`.tar+zstd`, or Docker's `application/vnd.docker.image.rootfs.diff.tar.gzip`. **Recommendation: encode deltas as `tar+gzip`** — spec makes `tar`/`tar+gzip` MUST-support but `tar+zstd` only SHOULD-support, so gzip maximizes cross-lane compat. The correctness guarantee for resume: "The final filesystem layout MUST match the result of applying the layers to an empty directory" ([manifest.md](https://github.com/opencontainers/image-spec/blob/main/manifest.md)) — this binds a *conformant applier* to reproduce the same rootfs from the *same layer content*; it is the correctness contract on apply, not a free property of any two nodes, and says nothing about the capture side producing equivalent tars.

**Loading on the target: one importer ingests both lanes.** On CRI, containerd's `images/archive` exposes `ImportIndex` that "implements Docker v1.1, v1.2 and OCI v1 … prefers OCI v1 … creates OCI index for Docker formats" ([pkg.go.dev images/archive](https://pkg.go.dev/github.com/containerd/containerd/images/archive)). One code path ingests Docker-format and OCI v1 bundles, normalizing the former. Two precisions: (1) `ImportIndex` ingests an image **tar bundle** (manifest/index + config + layer blobs), **not** a bare layer tarball; (2) in containerd v2 the path is `.../v2/core/images/archive` (v1 path valid for 1.x). To wrap an out-of-band delta, `WithManifest(manifest, names...)` attaches an already-built manifest and assigns ref names. `WithManifest` does **not** synthesize a manifest from a raw layer — the caller must first build the OCI manifest (config + delta descriptors), write manifest + blobs into the content store, then hand the descriptor to the exporter. Flow: build manifest → stage blobs → `WithManifest`. On the Docker lane, `docker load` ingests the equivalent archive. A documented hazard: Docker **v25.0.0-rc.1**'s `docker save` emitted an OCI archive with an empty `"layers": []` array ([moby#47065](https://github.com/moby/moby/issues/47065)) — a **transient RC regression**, P0, fixed (PR #47068) before v25.0.0 final, so **not** a current limitation. Its standing value: the manifest's layers array can be independently populated/emptied by tooling — verify the delta descriptor is present and its digest matches before trusting an imported archive. A claimed detail — that `Import`/`Export` operate only on an OCI-layout tar stream via `io.Reader`/`io.ReadCloser` with specific signatures — **did not survive verification**; treat the exact streaming signatures as unverified and confirm against the pinned containerd version.

**Cross-lane portability: where convergence is NOT free (open risk).** Decision (4) requires both lanes to converge on the *same* artifact, lane-specific capture code permitted. The apply-side spec guarantees a conformant applier reproduces the rootfs from given layer content — but does **not** guarantee two *capture* paths emit interchangeable tars. `docker commit` (overlayfs → OCI) and containerd `diff` can legitimately differ on: whiteout/opaque-dir encoding; xattr namespaces (`user.*`/`trusted.*`/`security.*`); timestamps and hardlinks (non-determinism breaks digest equality); and **uids inside the tar under differing userns mappings** (the critical one). None of these axes is backed by a verified primary claim here — derived from the apply-contract's *scope limits* and general overlayfs/OCI behavior. **Treat cross-lane interchange as an explicit validation milestone, not a settled property.** Capture code on *both* lanes must **normalize to a single OCI whiteout/xattr encoding**; "lane-specific capture OK" in decision (4) is precisely the license to do that normalization, and it is load-bearing. **userns uid-shift correctness — highest unverified risk.** Files the agent creates land on the host at shifted high uids; capture must record the *in-container* (namespaced) uid/gid in the tar, not the host-mapped uid — otherwise a delta captured under node A's mapping applies with wrong ownership under node B's mapping. No claim survived verification; **no spec citation**, only the architectural requirement. Must be designed and tested directly (capture relative to the namespace's id-map, or store in-container ids and re-shift on apply).

**Kopia fit: ship the layout, dedup the tars (thin evidence).** The OCI layout is "designed to be used in a variety of different transport mechanisms: archive formats (e.g. tar, zip), shared filesystem environments (e.g. nfs), or networked file fetching" ([image-layout.md](https://github.com/opencontainers/image-spec/blob/main/image-layout.md)) — spec-sanctioned support for shipping through Kopia→Garage rather than a registry. Precision: this applies to the **self-describing OCI image layout** (oci-layout + index.json + blobs/ + config + manifest), *not* a naked delta layer tar (just one blob, not self-describing). So **journal the OCI layout (or at minimum manifest + config + delta blobs), not a lone layer tar**, and keep the base pinned/local. Beyond "supported transport," the specific Kopia questions — CDC/dedup on a compressed tarball vs unpacked tree, size accounting — are **not** answered by any verified claim. A real tension: gzip-compressed tars defeat content-defined chunking (a small change near the front perturbs all downstream compressed bytes → poor dedup across delta generations). If migration dedup matters, consider journaling as **uncompressed `tar`** (still MUST-support) letting Kopia handle compression+CDC, or journaling the unpacked tree and reconstructing the tar deterministically on resume. **Unmeasured design choice — validate empirically.** Generation fencing inherits the existing monotonic-generation + DB-claim mechanism; the delta upload must be fenced on the same generation as the data-mount journal so a stale node cannot publish a delta resurrecting an evicted spawn.

**Node-side pinning and GC (mechanism known, citations thin).** Decision (3) requires the pinned base digest to stay alive for the spawn's life and be reclaimable on delete, on **both** lanes. Standard mechanisms are well known — containerd uses **leases** to pin content/snapshots against GC; Docker uses image reference holds / the containerd-backed image store's equivalents — but **no lease-API claim survived verification**, so the specific API shape is asserted only as the conventional approach. Design intent: take a lease (or equivalent ref) keyed to the live spawn referencing the pinned base manifest + layer blobs; release on delete to make the digest GC-eligible; never GC a digest with a live lease. Confirm the exact lease API against the pinned containerd version and Docker's hold semantics against the configured image store.

**Summary.** *Settled (high confidence, primary spec):* artifact = OCI image layout; base at `layers[0]`, deltas appended; `tar+gzip` for compat; digest-keyed blobs; transport over the journal is spec-sanctioned; `ImportIndex` + `WithManifest` ingest/produce both lanes' bundles; the v25 empty-layers bug is fixed. *Open / must validate (low confidence):* cross-lane capture normalization, **userns uid-shift correctness (highest risk)**, Kopia CDC/dedup behavior, exact lease/pinning APIs.

### S4. Fallback — engine-native upperdir preservation

This fallback splits into two mechanisms sharing an idea (let the engine keep the writable layer) with very different cost/risk: **Same-node suspend = stop-but-keep** (`docker stop`/`start` or a CRI stopped container with the sandbox kept leaves the writable layer in place; nothing captured/copied/interpreted) and **Migration = harvest-the-upperdir** (read the snapshotter's writable tree off the host and journal *that tree* through Kopia → Garage, reconstruct on node B). The first is cheap and attractive; the second fights the rest of the design and is strictly inferior to the decided OCI-diff capture for migration.

**Where the upperdir actually lives (mechanics well-pinned).** For the CRI/runsc lane, overlayfs is containerd's **default** snapshotter ([containerd snapshotters README](https://github.com/containerd/containerd/blob/main/docs/snapshotters/README.md)). Paths are stable: the active snapshot's upperdir is `<root>/snapshots/<id>/fs`, workdir `<root>/snapshots/<id>/work`, where `<root>` is e.g. `/var/lib/containerd/io.containerd.snapshotter.v1.overlayfs` ([overlay.go `upperPath`/`workPath`](https://github.com/containerd/containerd/blob/d939b6af5f8536c2cae85e919e7c40070557df0e/plugins/snapshots/overlay/overlay.go)). For an active snapshot with parents, `mounts()` emits a real `overlay` mount with `upperdir=<id>/fs`, `workdir=<id>/work`, `lowerdir=` parents joined by `:`. So "find the delta on disk" is easy. The hard part: this raw tree is **not** an OCI changeset — it carries char-device whiteouts, opaque markers, and overlay-private xattrs. Producing a portable layer still requires the snapshotter's differ to translate it.

**Why journaling the raw upperdir through Kopia is fragile.** Two classes of special metadata encode deletions/masking: (1) **Char-device whiteouts** — a deleted file is a char device 0/0 ([kernel overlayfs](https://docs.kernel.org/filesystems/overlayfs.html); [kmxz/overlayfs-tools](https://github.com/kmxz/overlayfs-tools)); recreating one on restore needs `mknod(S_IFCHR, makedev(0,0))` = **CAP_MKNOD**. (2) **Overlay xattrs** — opaque dirs marked `*.overlay.opaque=y`; redirect/metacopy/origin/impure also xattrs. The xattr namespace is the load-bearing trap and differs by lane: *Host/init-userns overlay (today's rootful Docker, containerd on host):* markers under `trusted.overlay.*`, readable/writable **only with CAP_SYS_ADMIN in the *initial* user namespace** ([xattr(7)](https://man7.org/linux/man-pages/man7/xattr.7.html)). This bites a Kopia *restore*: restic/Kopia-style restorers report restoring `trusted.*`/`system.*` xattrs "will fail as we don't have the CAP_SYS_ADMIN capability" — **even running as UID 0** ([restic#5089](https://github.com/restic/restic/issues/5089)), mirroring Spawnery's `--cap-drop=ALL` uid-0 posture. The kernel gate uses `capable()` (init_user_ns), **not** `ns_capable()`, so Decision #2's namespaced-root CAP_SYS_ADMIN would still **not** let you restore `trusted.*` xattrs. *Non-initial userns / rootless overlay:* on kernel ≥ 5.11 the mount *must* use `userxattr`, switching metadata to `user.overlay.*` ([moby#42177](https://github.com/moby/moby/issues/42177); [containerd#5076](https://github.com/containerd/containerd/pull/5076)); containerd auto-appends `userxattr` when `overlayutils.NeedsUserXAttr(root)` reports it. `user.overlay.*` *are* owner-readable/writable without CAP_SYS_ADMIN — so the userns case is *easier*, but only if harvest/restore detects which namespace is in use per mount instead of assuming `trusted.*`. Net: a naive userspace tree copy is the wrong tool — overlayfs-tools warns it "Only work[s] for regular files and directories… Do not use it on OverlayFS with device files" and "extended attributes might be lost" ([kmxz/overlayfs-tools](https://github.com/kmxz/overlayfs-tools)).

**The runsc-lane gotcha (decisive for the CRI lane).** Even though containerd prepares an overlay upper, runsc/gVisor **does not write the live delta there** during a running container. runsc interposes its own internal overlay and holds writes in a sandbox-internal (tmpfs/self-backed) upper; those writes are not reflected in the containerd host upperdir until gVisor flushes on stop/checkpoint. So harvesting `<root>/snapshots/<id>/fs` mid-flight gives a *stale* delta. You would have to stop/flush first anyway — at which point the snapshotter-agnostic containerd `diff` API (Decision #1) is the right and only reliable mechanism.

**Conflict with recreate-per-generation fencing.** "Stop-but-keep" and "harvest in place" both assume a *persistent* engine-managed container/snapshot whose identity survives the suspend boundary — in direct tension with recreate-per-generation (the new generation gets a fresh pod and, on CRI, a fresh sandbox/snapshot id). Same-node suspend can paper over this only if the node deliberately retains the prior generation's stopped container/snapshot and re-binds it.

**Verdict vs. the layers approach (Decision #1).** For **migration**, upperdir-harvesting is strictly worse: `docker commit` / containerd `diff` run **inside the privileged daemon** (holds CAP_SYS_ADMIN) and **normalize** overlay whiteouts/opaque into portable OCI `.wh.` tar whiteouts — sidestepping the char-device + `trusted.overlay.*`/`user.overlay.*` minefield and producing the single convergent artifact (Decision #4). A raw Kopia upperdir walk produces a lane-specific, privilege-sensitive artifact and re-introduces the CAP_SYS_ADMIN-on-restore problem cap-dropped restorers provably cannot satisfy ([restic#5089](https://github.com/restic/restic/issues/5089)). The **only** place upperdir preservation wins is the narrow **same-node suspend/resume** case, in its "stop-but-keep" form: zero capture cost — no `docker commit`, no `diff`, no Kopia round-trip, no whiteout/xattr handling. **Recommendation: keep "stop-but-keep" as the zero-cost same-node suspend path where the node can retain engine state across the generation boundary, but do *not* use raw upperdir-harvesting as a migration mechanism.** Raw upperdir harvesting is, at best, a debugging/last-resort escape hatch.

### S5. Package-manager reality check

The load-bearing finding: **the namespaced full cap set is sufficient for apt/dpkg/useradd/chown/chmod, but only over UIDs/GIDs mapped into the spawn's idmap range** — the real work is in idmap and mount-flag policy, not adding capabilities.

**S5.1 What apt actually does as root.** Since APT 1.1 (2015) `apt` drops privilege to the unprivileged `_apt` user to sandbox download workers, calling `seteuid`/`setegid`/`setgroups` (needs `CAP_SETUID`, `CAP_SETGID`) ([Debian #813786](https://bugs.debian.org/cgi-bin/bugreport.cgi?bug=813786); [capabilities(7)](https://man7.org/linux/man-pages/man7/capabilities.7.html)); it also `chown`s download/partial dirs to `_apt:_apt` (needs `CAP_CHOWN`). Crucially, **this privilege-drop is best-effort, not mandatory.** Per maintainer Julian Andres Klode: *"If you fetch in a directory `_apt` cannot write to, it will run the fetchers as root instead of `_apt`"* ([#813786](https://bugs.debian.org/cgi-bin/bugreport.cgi?bug=813786)). When the drop can't happen (missing `CAP_SETUID`, today's cap-drop-ALL agent), apt continues, fetching as root with `W: Download is performed unsandboxed as root…`. **Spawnery implication:** under today's `--cap-drop=ALL`, `apt install` of a normal package often still *succeeds* (just unsandboxed), because dpkg-side writes happen as in-container root into the rootfs. The failures the design chases come from operations needing namespaced privilege over the rootfs — and the warning path is a security downgrade we'd rather not run in. Scope limit: the root-fallback covers **only** apt's own Acquire download sandbox, **not** dpkg unpacking or maintainer-script writes (those run as in-container root, governed by dropped capabilities).

**S5.2 The userns cap set vs. what apt/dpkg require — verified sufficient.** A container init created via `clone()/unshare()` with `CLONE_NEWUSER` (runc, rootless Podman, runsc) starts with a **complete capability set in the new namespace** ([user_namespaces(7)](https://man7.org/linux/man-pages/man7/user_namespaces.7.html)). Mapping:

| apt/dpkg operation | Required capability | In namespaced full set? |
|---|---|---|
| Drop to `_apt` (`seteuid`/`setegid`/`setgroups`) | `CAP_SETUID`, `CAP_SETGID` | Yes |
| `chown partial/` → `_apt:_apt` | `CAP_CHOWN` | Yes |
| Maintainer scripts: `chmod` of non-owned files | `CAP_FOWNER` (+ `CAP_DAC_OVERRIDE`) | Yes |
| `useradd`/`addgroup` (write `/etc/passwd,shadow,group`; chown home) | `CAP_DAC_OVERRIDE` + `CAP_CHOWN` | Yes |

The file-permission-bypass quintet `CAP_CHOWN, CAP_DAC_OVERRIDE, CAP_DAC_READ_SEARCH, CAP_FOWNER, CAP_FSETID` is explicitly enumerated in [user_namespaces(7)](https://man7.org/linux/man-pages/man7/user_namespaces.7.html). Precision: **`useradd`/`addgroup` do not fundamentally need `CAP_SETGID`** — their core work is DAC-writing account files + chown home (`CAP_DAC_OVERRIDE` + `CAP_CHOWN`); `CAP_SETGID` only matters when a maintainer script itself drops to another uid/gid. **Conclusion:** the userns full namespaced set satisfies every requirement; no custom cap list needed.

**S5.3 The binding constraint is the idmap, not the cap list.** Namespaced capabilities are *"valid only for operations on objects governed by that namespace"* — a capability is effective on a file only when **the file's owner UID/GID are mapped into the namespace** ([user_namespaces(7)](https://man7.org/linux/man-pages/man7/user_namespaces.7.html)). Files owned by unmapped host UIDs appear as overflow (`nobody:nogroup`) and remain immutable even with `CAP_CHOWN`/`CAP_DAC_OVERRIDE`/`CAP_FOWNER`. Three consequences: (1) **Rootfs/delta: fine** — operations on in-map uids land in the writable layer the delta-capture harvests. (2) **Data mounts (`/app/<path>`): must be owned within the map** (or world-accessible); with CRI/runsc prefer **idmapped mounts**. The existing world-writable data-mount work (sp-ei4.1.1) and userns idmap must be designed together. (3) **Use a multi-range idmap, not a 1:1 map** — with only one mapped id, apt's `setgroups`/`setegid` to `_apt` hits `EPERM` ([Apptainer userns docs](https://apptainer.org/docs/admin/main/user_namespace.html), corroborating [user_namespaces(7)](https://man7.org/linux/man-pages/man7/user_namespaces.7.html)). *(Confidence: medium on the specific apt-EPERM-on-single-map; well-established in rootless practice.)* The **secrets tmpfs** is unaffected except for ownership: owned within the map so the agent reads it, and — independently of userns — kept out of the delta.

**S5.4 nosuid/noexec and setuid binaries in the delta.** Two distinct mechanisms neutralize a setuid binary in the delta: (a) **Userns unmapped-owner** — a setuid bit is *"silently ignored"* if **either** the file's owner uid or gid has no mapping, *"mirror[ing] the semantics of MS_NOSUID"* ([user_namespaces(7)](https://man7.org/linux/man-pages/man7/user_namespaces.7.html)). But for a **root-owned** binary with uid 0 **mapped** (normal case), the bit is honored. So userns does **not** by itself defang an in-rootfs root-owned `sudo`. (b) **MS_NOSUID mount flag** — if the mount is `nosuid`, the bit is ignored regardless of mapping; this is the lever to actually disable setuid binaries in a mount. Implications: setuid binaries apt installs into the rootfs work under userns (good for dev-parity, captured cleanly); if `/app` is `nosuid`/`noexec`, a user-installed setuid/exec there won't gain setuid/exec (deliberate, acceptable, must be documented); both lanes converge (kernel-level, identical under runc and runsc).

**S5.5 Other installers (brief, lower-confidence).** By reasoning (**not** separately verified): `npm -g`, `pip --user`/venv, `cargo install` write into user-owned prefixes with no privilege transition, so they work today as in-container root; userns only helps global-prefix (`/usr/local`) cases via namespaced `CAP_DAC_OVERRIDE`. `curl|sh` varies — those that `sudo`/write `/usr` benefit; those needing genuine host privilege still can't (by design). **Low confidence — spot-check in e2e.**

**S5.6 sudo + non-root user, or root-by-default?** Once userns makes in-container root namespaced-unprivileged ([Apptainer](https://apptainer.org/docs/admin/main/user_namespace.html); [user_namespaces(7)](https://man7.org/linux/man-pages/man7/user_namespaces.7.html)), "root inside" is no longer a host-trust concern — the security argument for a non-root default user largely dissolves. The remaining reason for `sudo` + non-root user is **dev-parity**. **Recommendation: stay root-by-default for the agent image, rely on userns + a populated idmap for safety; add `sudo`/non-root user only behind a per-app opt-in where dev-parity demands it.** *(Confidence: low on the parity tradeoff.)*

**S5.7 Net for the delta-capture / migration design.** Nothing in the package-manager path threatens the delta-as-OCI-layer model — apt/dpkg/useradd/chown/chmod all write into the writable layer, which is exactly what `docker commit` / containerd `diff` capture. The two things to get right are **policy, not capture**: (1) the spawn's **idmap** must be a populated multi-range map with data-mount ownership inside it (idmapped mounts on CRI); (2) the **mount-flag posture** (`nosuid`/`noexec` on data mounts, normal on rootfs) must be deliberate and documented.

### S6. Prior art — who persists agent/dev-env rootfs writes

The headline: **our chosen design has a direct, documented precedent in Modal's filesystem snapshots**, while E2B/Firecracker (memory/VM snapshots) and Codespaces (data-only dev-env) bracket the two failure modes we steer between.

**Modal — filesystem snapshots (closest analogue).** Modal's `Sandbox.snapshot_filesystem()` produces a **delta** of the rootfs: snapshots "are calculated as the difference from your base image, so only modified files are stored" ([modal.com/docs/guide/sandbox-snapshots](https://modal.com/docs/guide/sandbox-snapshots)); the Directory Snapshots blog independently describes them as "diffs from your base image" ([modal.com/blog/directory-snapshots](https://modal.com/blog/directory-snapshots-resumable-project-state-for-sandboxes)). Delta granularity, not full-disk copy — same as our `docker commit` / snapshot-diff capture. Crucially, the snapshot is **itself a reusable image**: "These Snapshots are Images and can be used to create new Sandboxes" (`image = sb.snapshot_filesystem(); sb2 = modal.Sandbox.create(image=image, ...)`) — precisely our "resume = pinned-base + delta, both lanes converge on one portable artifact" contract, in another vendor's product. Scoping caveat: Modal also exposes a **separate** memory-snapshot type (full RAM+filesystem) — the E2B-style stateful capture we exclude. "Snapshot" is overloaded; Modal shipping both side-by-side reinforces that filesystem-delta and memory-state are distinct products, and picking filesystem-delta-only is a coherent, vendor-validated point. *(Confidence: high — vendor primary docs + blog; no published byte-pricing/quota figures found — open.)*

**E2B / Firecracker — full memory+disk snapshots (the cost we avoid).** E2B `pause` is **stateful**: "both the sandbox's filesystem and memory state will be saved" ([e2b.dev/docs/sandbox/persistence](https://e2b.dev/docs/sandbox/persistence)). A Firecracker microVM snapshot, not a delta-only disk capture. Benefit: a resumed sandbox skips setup (reloading ML models into RAM); cost: **pause latency ~4 seconds per 1 GiB of RAM** ([e2b.dev/docs/sandbox/persistence](https://e2b.dev/docs/sandbox/persistence)) — the dominant tax our suspend-based data-only design refuses. Underlying Firecracker mechanics carry three warnings reinforcing our choices: (1) **Diff snapshots are only approximately delta** — they "save at least the current microVM state and the memory accessed since the last snapshot … in a sparse file" and "might include more pages than strictly needed" ([firecracker snapshot-support.md](https://github.com/firecracker-microvm/firecracker/blob/main/docs/snapshotting/snapshot-support.md)); our file-level delta is exact by construction. (2) **Restore is asymmetric — the engine does not apply deltas at resume** — Firecracker "will still only restore full snapshots, so users will need to merge incremental snapshots externally" ([issue #2142](https://github.com/firecracker-microvm/firecracker/issues/2142)); for us that "merge" is `pinned-base + delta-layer` performed by the runtime layer stack at pod recreate — far cheaper. (3) **VM snapshots are portability- and security-fragile** — each binary fixes one format version; cross-host-kernel restore ships an explicit "we do not provide any guarantees" disclaimer; cloning a memory snapshot reuses entropy/RNG seeds/one-time tokens; integrity is only a 64-bit CRC ([snapshot-support.md](https://github.com/firecracker-microvm/firecracker/blob/main/docs/snapshotting/snapshot-support.md)). These are *properties of memory-state snapshots* we exclude — an OCI layer tar is host-kernel-agnostic and carries no live RNG/entropy state. *(Confidence: high — upstream primary docs.)*

**GitHub Codespaces — data-only persistence (the failure mode we avoid).** Only `/workspaces` (the cloned repo) is durably preserved across **both** stop/start and container rebuild ([GitHub Codespaces docs](https://docs.github.com/en/codespaces/developing-in-a-codespace/persisting-environment-variables-and-temporary-files)). Everything outside is "tied to the lifecycle of the container" — survives stop/start but is **discarded on rebuild** (narrow exception: `/tmp` survives rebuild but clears on stop). Concretely, **apt-installed packages (`/usr`, `/var`) and home-directory edits do not survive a rebuild** ([persisting docs](https://docs.github.com/en/codespaces/developing-in-a-codespace/persisting-environment-variables-and-temporary-files); [rebuilding docs](https://docs.github.com/en/codespaces/developing-in-a-codespace/rebuilding-the-container-in-a-codespace)). This is exactly the gap Spawnery's delta capture closes: our journal is data-mount-only (the `/workspaces` equivalent), and an in-container-root agent will `apt install`, `chown`, and write to `/etc`/`/usr`/home — mutations a Codespaces-style model loses on every rebuild. *(Confidence: high — first-party docs, two corroborating pages.)*

**Image upgrades under preserved deltas.** Surveyed platforms **resolve this by tying the delta to the base it was taken from**: Modal snapshots are "the difference from your base image," meaningful only relative to that specific base; Codespaces sidesteps the question (a rebuild can pull a newer base and discards the rootfs delta). None of the surveyed vendors document an automatic rebase of a captured rootfs delta onto a *newer* base. This directly supports our **per-spawn digest pinning**: a captured delta is coherent only against the exact base it diverged from, so pin the base by digest for the spawn's life and apply image upgrades to *new* spawns only. The corollary — nodes must not GC a pinned base while a live spawn references its delta — is the local-image analogue of Modal keeping the parent Image addressable. *(Confidence: high on "deltas are base-bound"; medium on absence of any delta-rebase feature — evidence-of-absence.)*

**Synthesis.** **Modal proves the mechanism** (rootfs divergence as a base-relative delta, materialized as a reusable image — a shipped product). **E2B/Firecracker price the rejected alternative** (full memory+disk ~4s/GiB-RAM + cross-kernel fragility + entropy reuse). **Codespaces names the gap we close** (data-only loses apt/chown/rootfs on rebuild). Two lanes converging on one OCI-layer artifact, shipped delta-only through Kopia→Garage, is the synthesis: Modal's delta-as-image portability without E2B's memory tax, closing the Codespaces rootfs gap, with digest pinning resolving the upgrade-vs-preserved-delta tension the way every surveyed vendor implicitly does.

### S7. Security delta

An honest before/after: moving the agent from *root + `--cap-drop=ALL`, no userns* to *root-in-userns with a generous in-userns capability set*, plus the secrets-leakage and disk-exhaustion surfaces delta capture creates. Lanes are separated because their kernel-LPE threat models differ structurally.

**S7.1 Sandbox strength: cap-drop-ALL vs. root-in-userns (runc lane).** The motivation for userns is operational, not security. The honest answer: **neither configuration removes the host kernel from the syscall path, so both remain exposed to kernel LPE; userns is double-edged.** (a) **The userns feature has a poor LPE track record, and enabling it is frequently the *precondition* for exploitation** — CVE-2024-1086 (`nf_tables` double-free, v5.14–v6.6) "depends on the kernel's unprivileged user namespaces feature accessing `nf_tables`," "enabled by default on Debian, Ubuntu" ([CrowdStrike](https://www.crowdstrike.com/en-us/blog/active-exploitation-linux-kernel-privilege-escalation-vulnerability/); PoC requirements list `CONFIG_USER_NS=y` + `kernel.unprivileged_userns_clone=1` + `CONFIG_NF_TABLES=y`). (b) **But the runtime-configured userns the design proposes is not the same as unprivileged-userns-clone-from-inside** — the pod runs in a userns set up *by the node*, mapping in-container uid 0 to an unprivileged high host uid; the agent's "generous" caps are **namespaced**. That mapping (container-root ≠ host-root) is defense-in-depth the cap-drop-ALL-but-shared-userns posture lacks today; the residual exposure is the kernel's *userns code paths* being reachable. (c) **Caveat (under-evidenced):** whether today's cap-dropped agent can *already* `unshare` a nested userns depends on `kernel.unprivileged_userns_clone`, the seccomp profile, and (Ubuntu 23.10+/24.04) an AppArmor LSM gate. We did not verify Spawnery's current effective state. If unprivileged userns is already reachable, the change adds little *new* surface; if blocked, it adds it. **Must be measured on target node images before accepting.** Net: treat userns as operationally necessary but kernel-LPE-neutral-to-slightly-negative; compensate with host controls (keep `kernel.unprivileged_userns_clone`/`user.max_user_namespaces` and the seccomp allowlist tight, patch netfilter CVEs), and **do not run untrusted multi-tenant workloads on the runc lane at all.**

**S7.2 How `runsc` changes the cloud-lane threat model.** Under gVisor `runsc` the agent does not talk to the host kernel API at all — the strongest, best-sourced result here: "No system call is passed through directly to the host. Every supported call has an independent implementation in the Sentry" ([gVisor security architecture](https://gvisor.dev/docs/architecture_guide/security/)); "Sandboxed programs simply can't make system calls directly to the host in the first place" ([Systrap release](https://gvisor.dev/blog/2023/04/28/systrap-release/)). The agent interacts with a *virtualized* System API, so "even with appropriate capabilities, a user in a gVisor sandbox will only be able to manipulate virtualized system resources … and not underlying host system resources" ([gVisor](https://gvisor.dev/docs/architecture_guide/security/)). Consequence: **unprivileged-userns CVEs and the broader host-kernel syscall-bug class are not *directly* reachable by the agent on the cloud lane** — a CVE-2024-1086-style `nf_tables` bug lives in the host kernel; the agent's `nft`/`unshare`/socket syscalls are serviced by the Sentry's reimplementation. Two precision caveats: `runsc` reduces but does not eliminate host-kernel surface (the Sentry issues a small seccomp-allowlisted set of host syscalls — ~53 without networking, ~211 with; no `execve`/`open`/socket-creation — "radically reduc[ing]" but not zeroing it; a Sentry compromise remains a mitigated residual vector) ([gVisor Security Basics](https://gvisor.dev/blog/2019/11/18/), [security architecture](https://gvisor.dev/docs/architecture_guide/security/)); and `runsc` itself runs the workload in an isolated userns with minimal caps, so it is a *superset* of the runc-lane userns isolation — the "runsc > runc+userns against kernel LPE" direction is sound.

**S7.3 Egress floor under userns and across lanes.** (a) **netns and userns are orthogonal** — adding a userns does not move/rename the pod's netns or interfaces; the spawnlet applies the floor from outside the agent's privilege domain, so *application* of the floor is unaffected (reasoned; medium confidence). (b) **Integrity caveat — netns ownership vs. in-userns `CAP_NET_ADMIN` (important, under-cited):** the agent and sidecar share one netns, and "generous in-userns caps" plausibly include `CAP_NET_ADMIN`. If the pod netns is *owned by the agent's userns*, in-userns `CAP_NET_ADMIN` would let the agent flush/rewrite the egress rules and defeat the floor. Floor integrity requires the enforcing netns **not** be owned by the agent's userns — enforce in a parent/host netns, have the node own the netns in a userns the agent is not root in, or anchor egress control in the sidecar/host path. On the `runsc` lane this is moot for host-kernel netfilter (the agent's `iptables` hits the Sentry). (Low/medium confidence — open verification item.) (c) **Rootless (future Podman) breaks bridge-keyed enforcement entirely** — slirp4netns/pasta egress "is exposed in the host as a normal process and not a link" ([podman#18715](https://github.com/containers/podman/discussions/18715); [slirp4netns](https://github.com/rootless-containers/slirp4netns)); host iptables keyed on a bridge/veth cannot match it. The floor must move to per-netns enforcement or host filtering by owner-uid/cgroup. (High confidence on mechanism; rootless is future work.)

**S7.4 Secrets leakage into the delta.** (a) **The secrets tmpfs itself does not leak through the delta** — a separate bind-mounted mount, not part of the writable upper, so not captured by `docker commit` / snapshot diff (high confidence on the mechanism; confirm with a capture-time test). (b) **The real residual is agent-copied secrets** — if in-container (now-userns) root copies a secret *out of* the tmpfs *into* the rootfs, the copy is in the delta and journaled. A genuine new exfiltration-to-journal surface. (c) **Mitigations are design-internal, not externally verifiable** (flagged as such): owner-sealed Kopia repo keys (the Garage operator cannot read journaled deltas — bounds *who* sees a leaked secret, not whether it is captured); capture-time exclusion/denylist of known-sensitive dirs; documented acceptance that an agent which deliberately root-copies its own secret into its own rootfs is within its own trust boundary (it could already exfiltrate over its own egress subject to the floor). The spec should state this acceptance explicitly rather than claim leakage is prevented; exclusion lists are best-effort.

**S7.5 Disk bombs: bounding delta growth.** (a) **The CRI/`runsc` lane has no default quota on the writable layer** — containerd's default overlayfs snapshotter "is not limited now, so when user writes lots data to container's root, it will cause the host's root disk full" ([containerd#3329](https://github.com/containerd/containerd/issues/3329), dup of open [#759](https://github.com/containerd/containerd/issues/759)); quota is "tied to the specific implementation of the Snapshotter being used," the cross-snapshotter abstraction unresolved. (High confidence.) (b) **Bounding growth requires a snapshotter/backing-FS-specific mechanism**: XFS/ext4 project quota (`prjquota`) on the snapshotter's backing dir; or an alternate snapshotter capping the upper natively (devmapper thin-device sizing, EROFS block-mode fixed-size upper); or `runsc`'s own overlay sizing. Pick one per node image as part of provisioning. (c) **Docker lane `--storage-opt size=` support is backing-driver-specific and was *not* verified** — widely understood to require overlay2-on-xfs-with-pquota or devicemapper, unsupported on overlay2-over-ext4 without project quota, but no verified citation. **Treat the matrix as open; confirm against current Docker/Moby docs.** (Low confidence.) (d) **Behavior at the cap is a design choice** — (a) hard-fail writes (ENOSPC, can wedge mid-task) or (b) suspend-and-warn (detect approaching the cap, trigger suspend/journal). Recommendation: enforce a hard quota as backstop (DoS containment non-negotiable) and layer a softer suspend-and-warn threshold below it.

**S7.6 Bottom line.** Cloud/CRI lane under `runsc` is the stronger sandbox against kernel LPE by construction — route untrusted multi-tenant tenants here (high confidence). runc-lane userns is operationally required and kernel-LPE-neutral-to-slightly-negative; verify current effective userns/seccomp state before rollout (open). Egress-floor integrity under userns hinges on the pod netns *not* being owned by the agent's userns; rootless lanes need a non-bridge strategy. Baseline secrets stay out of the delta; agent-copied secrets are an accepted, owner-sealed-mitigated residual. Delta growth is unbounded by default and must be capped per-lane; Docker's `--storage-opt` matrix needs confirmation.

## 5. Concrete recommendation for Spawnery

**Mechanism per lane (capture).**
- **Docker/runc lane (today, single-tenant):** capture the writable-layer delta with **`docker commit`** at the default pause (do *not* pass `--no-pause` — the freeze *is* the quiesce window). Data mounts and the secrets tmpfs are provably excluded by the Engine storage model. Squash the overlay chain before ~120 stacked deltas.
- **CRI/containerd + runsc lane (cloud, multi-tenant):** capture via the **containerd snapshot `diff`** Go API in the `k8s.io` namespace. Gated on validating: snapshotter-key discovery from the CRI container id, stopped-but-not-removed lifecycle, diff-service privileges, and — decisively — whether `runsc --overlay2` writes are visible to the host snapshotter or require a stop/flush first.

**How they share one artifact.** Both lanes **normalize to a single self-describing OCI image layout** (`blobs/` + `oci-layout` + `index.json` + synthesized config + manifest), base at `layers[0]`, deltas appended in capture order, layer media type `tar+gzip`. Lane-specific *capture code* is permitted (decision #4) precisely so it can normalize whiteout/opaque-dir encoding, xattr namespaces, timestamps/hardlinks, and — highest risk — **userns uid-shift** (record in-container namespaced uids/gids in the tar, re-shift on apply) to one encoding that applies correctly on either lane. Journal the **OCI layout** (or at minimum manifest + config + delta blobs), never a lone layer tar, through the existing Kopia→Garage path, fenced on the same monotonic generation as the data-mount journal. Same-node suspend keeps the delta **locally** (prefer zero-cost "stop-but-keep" where the node can retain engine state across the generation boundary); only migration ships the delta.

**Userns / ownership policy.** Both lanes: a **populated multi-uid subid range** (≈65536-wide), **`setgroups` permitted**. Docker: daemon-wide `userns-remap` (gets the wide range for free, loses per-spawn isolation). CRI: KEP-127 plumbing on a containerd build past the #10611 `setgroups`-deny regression (or verify `setgroups=allow` at node bring-up). Ship `APT::Sandbox::User "root"` in the base image as belt-and-suspenders. Writable `/app` mounts + secrets tmpfs need ownership in-map: prefer **idmapped mounts** (CRI lane especially) over chown-into-range; 0777 still works but is the weakest. Read-only `/app` mounts need no special handling. Stay **root-by-default** for the agent image (userns dissolves the host-trust argument); add `sudo`/non-root user only behind per-app opt-in.

**Pinning + GC.** Pin the base by **digest** for the spawn's whole life; image upgrades apply to *new* spawns only (no delta-rebase onto a newer base — no surveyed vendor does this). Hold a containerd **lease** (Docker: image-ref hold on the containerd-backed store) keyed to the live spawn, referencing the pinned base manifest + layer blobs; release on spawn delete to make the digest GC-eligible; **never GC a digest with a live lease.** Confirm exact lease/hold APIs against the pinned versions.

**Secrets & disk-bomb mitigations.** Secrets tmpfs stays a **separate mount** (excluded from the delta by construction — confirm with a capture-time test in Spawnery's concrete pod spec). Accept agent-copied-secrets-into-rootfs as an owner-sealed-Kopia residual; layer a best-effort capture-time denylist for known-sensitive dirs. **Mount junk/cache dirs (`/var/cache/apt`, build caches) as excluded volumes/tmpfs** so they fall outside the writable layer (reusing the proven exclusion). Enforce a **hard per-spawn delta quota** as DoS backstop (prjquota on XFS/ext4, or devmapper/EROFS-sized snapshotter, or runsc overlay sizing — pick one per node image) with a softer **suspend-and-warn** threshold below it. Re-anchor the **egress floor** so the enforcing netns is *not* owned by the agent's userns (and confirm whether the generous cap set actually includes `CAP_NET_ADMIN`/`CAP_NET_RAW`).

## 6. Staged adoption path

**Stage 0 — userns + unrestricted writable rootfs (operational fix).**
*Trigger:* `--cap-drop=ALL` failures (apt/dpkg/useradd/chown/chmod of non-owned files) are blocking agent workflows, and the 0777 data-mount workaround is degrading.
*Action:* Docker daemon-wide `userns-remap` (≥65536 range) on the single-tenant lane; CRI KEP-127 userns on a containerd build past the #10611 fix. Idmapped mounts (or chown-into-range) for writable `/app`; secrets tmpfs owned in-map. No capture yet.
*Exit threshold:* apt/chown/chmod "just work" and the security state (effective `unprivileged_userns_clone`/seccomp/AppArmor userns gate) has been measured on target node images.

**Stage 1 — same-node delta survival (local only).**
*Trigger:* spawns need to survive node-local suspend/resume (operator-initiated pause, node maintenance window).
*Action:* zero-cost **"stop-but-keep"** where the node can retain engine state across the generation boundary; otherwise local `docker commit` keeping the delta on disk. No journal round-trip. Define the squash trigger (flatten before ~120 stacked deltas) and the fencing-vs-engine-state reconciliation (can the node retain the prior generation's stopped container/snapshot without violating single-writer fencing?).
*Exit threshold:* deletion fidelity (deleted file + replaced opaque dir) round-trips a local suspend→resume, and the squash procedure is validated on target kernels (~122 mount-option page limit).

**Stage 2 — migratable deltas (cross-node, Docker lane first).**
*Trigger:* placement/rebalancing requires moving a live spawn between nodes (suspend on A → resume on B).
*Action:* `docker commit` → OCI image layout → Kopia→Garage (generation-fenced) → resume as pinned-base + delta on node B. Gated on validating **cross-lane-agnostic** OCI normalization and **userns uid-shift correctness** (the highest-risk unverified item), plus the Kopia storage-form choice (uncompressed tar vs gzip vs unpacked tree — measure dedup across delta generations) and exact lease/pinning APIs.
*Exit threshold:* a Docker-lane delta survives suspend→migrate→resume with correct ownership under a different node's id-map, and pinned-base GC protection holds.

**Stage 3 — multi-tenant cloud lane (CRI + runsc).**
*Trigger:* onboarding untrusted multi-tenant tenants to cloud nodes.
*Action:* CRI/containerd `diff` capture under `runsc`; validate that agent writes under `--overlay2` are host-snapshotter-visible (or define the stop/flush trigger and its gofer-I/O cost); confirm both lanes emit byte-compatible OCI artifacts so a delta is genuinely lane-agnostic on resume across lanes; enforce per-lane delta quota; re-anchor the egress floor outside the agent's userns.
*Exit threshold:* a delta captured on a runsc node applies cleanly on a Docker node (and vice versa), quota containment is proven, and egress-floor integrity holds under userns.

## 7. Weak-evidence / open questions & runsc-dominated callouts

**runsc-dominated callouts (the CRI/gVisor lane carries the most unverified risk).**
- **Does the host snapshotter ever see the writes?** With `runsc --overlay2`, agent writes are held in a sandbox-internal upper and not reflected in the containerd host upperdir until gVisor flushes on stop/checkpoint. Whether a host-side `containerd diff` can observe *any* delta mid-flight, or whether the in-sandbox overlay must be disabled/host-visible (and at what gofer-I/O cost), is **untested** and gates the entire CRI capture path.
- **Does runsc's Sentry-implemented userns honor namespaced caps for apt/chown/chmod identically to the kernel**, and does runsc-under-containerd participate in KEP-127 pod-userns plumbing? Unverified.
- **runsc internal-overlay flush/checkpoint trigger** determines whether "stop-but-keep" is even coherent on this lane. Unknown.
- Idmapped-mount availability/need on the runsc lane to keep data-mount ownership in-map without chowning host trees. Unverified.

**Highest-risk unverified items (no citation — design and test directly).**
- **userns uid-shift normalization:** does the delta tar record in-container namespaced uids/gids, and does apply re-shift correctly under a different node's id-map? Does idmapped mount make this transparent or is an explicit ownership-shift needed at restore? (Highest risk for cross-node migration.)
- **Cross-lane capture interchange:** do `docker commit` and containerd `diff` produce interchangeable tars (whiteout/opaque, xattr namespaces, timestamps, hardlinks) for the same delta, and what single OCI encoding should both normalize to?

**Open questions carried from the research pass.**
- Exact containerd version cutline where the #10611 `setgroups`-deny regression is fixed (PR #10741 → which release tag), to pin a known-good CRI build.
- Precise kernel + filesystem version requirements for idmapped bind mounts on Spawnery's target node OS.
- Can the non-Kubernetes CRI client (spawnlet) drive `UserNamespace`/`NamespaceMode` directly, with idmapped-mount support, on containerd 1.7 vs 2.x?
- Should the base image ship `APT::Sandbox::User "root"` as default, or rely solely on a correctly-sized userns?
- Benchmark `docker commit` wall-clock on realistic multi-GB agent deltas (the pause = the suspend window); does it require streaming/lazy capture?
- Validate the containerd Go diff path: snapshotter-key derivation from CRI container id, `k8s.io` namespace interop, stopped-not-removed requirement, diff-service privileges.
- Confirm both lanes emit byte-compatible OCI layer artifacts (same whiteout/opaque conventions).
- Decide/test a capture-time junk-exclusion strategy for `/var/cache/apt` and build caches (preferably excluded volumes/tmpfs).
- Round-trip deletion fidelity (deleted file + replaced opaque dir) across suspend→migrate→resume under userns uid mapping.
- Define the squash trigger/procedure for repeated same-node suspend; confirm the ~122 mount-option page limit on target kernels.
- Kopia storage form: uncompressed tar (better CDC dedup) vs gzip tar vs unpacked tree — measure dedup across successive delta generations.
- Exact pinning APIs: containerd lease create/cancel + label-based GC roots; Docker image-ref hold under the containerd-backed image store.
- Confirm actual containerd Import/Export signatures and whether they require an OCI-layout bundle vs accept file paths (the streaming-signature claim was refuted).
- For rootless-Podman/containerd: ensure no init-userns (`trusted.overlay.*`) harvest of an upperdir written under `userxattr` (`user.overlay.*`) — the documented overlayfs I/O-error trigger ([moby#42177](https://github.com/moby/moby/issues/42177)).
- Can the node retain the prior generation's stopped container/snapshot across the GENERATION fence for zero-cost "stop-but-keep" without violating single-writer fencing? Needs a concrete reconciliation design.
- Does the existing Kopia repo format round-trip char-device-0/0 inodes and per-namespace overlay xattrs, or silently drop them? Empirical snapshot/restore test before any upperdir escape-hatch is viable.
- Spot-check `npm -g`/`pip`/`cargo install`/`curl|sh` for any operation needing namespaced privilege beyond writing user-owned dirs (e2e, not assumed).
- What do comparable products (Coder, Gitpod/Ona, Codespaces, e2b, Daytona, Fly machines, Replit, Morph/Sprites) ship — root-by-default vs sudo+non-root — and how do they capture/restore rootfs writes? Several unsurveyed; possible follow-up pass.
- Modal filesystem-snapshot byte-pricing and size/quota limits; restore latency for Modal snapshots and Codespaces rebuild-with-restore (only E2B's ~4s/GiB was verified).
- Does any platform rebase a preserved rootfs delta onto a *newer* base (vs discarding on upgrade)? Found none — evidence-of-absence; confirm whether per-spawn pinning is the universal answer or merely common.
- Published postmortems for Modal filesystem snapshots / E2B disk persistence (corruption, partial-delta, base-image drift)? None surfaced.

**Security-state open items (measure before rollout).**
- Spawnery's current node-image effective state (`kernel.unprivileged_userns_clone` / `user.max_user_namespaces` sysctls, agent seccomp profile, AppArmor userns gate) — decides whether the proposed userns change adds new kernel-LPE surface or merely formalizes existing reachability.
- Will the pod netns enforcing the egress floor be owned by the agent's userns? If so, in-userns `CAP_NET_ADMIN` defeats the floor — re-anchor enforcement.
- Confirm whether the "generous in-userns cap set" actually includes `CAP_NET_ADMIN`/`CAP_NET_RAW` (that is what makes the egress-tampering concern live).
- Verify the exact Docker `--storage-opt size=` support matrix per graph driver / backing filesystem (overlay2+xfs/pquota vs ext4 vs devicemapper) before relying on it to cap Docker-lane delta growth.
- Lab-verify that `docker commit` and the containerd snapshot diff actually exclude the separately-bind-mounted secrets tmpfs from the delta in Spawnery's concrete pod spec.
- Decide/document the per-lane delta-quota implementation (prjquota vs alternate snapshotter vs runsc overlay sizing) and the at-cap behavior (hard-fail vs suspend-and-warn thresholds).

## Appendix A — Refuted claims (killed by adversarial verification)

For transparency: claims that did NOT survive 3-vote verification — do not assert these.

- **[S1]** (vote 1-2) In recent containerd 2.x versions using user namespaces, the setgroups syscall fails with EPERM from inside the constructed user namespace because /proc/self/setgroups is written as 'deny'.
  — source: https://github.com/containerd/containerd/issues/10742
- **[S3]** (vote 1-2) containerd's Import()/Export() operate on an OCI layout tar stream (oci-layout file + blobs/ directory), not the legacy Docker tarball format, and use stream-based I/O (io.Reader/io.ReadCloser) rather than file paths.
  — source: https://github.com/containerd/containerd/pull/1013
- **[S3]** (vote 0-3) The Export API returns a readable stream and Import accepts a reader: Export(ctx, ocispec.Descriptor, ...ExportOpt) (io.ReadCloser, error) and Import(ctx, string, io.Reader, ...ImportOpt) (Image, error).
  — source: https://github.com/containerd/containerd/pull/1013
- **[S5]** (vote 0-3) Creating a user namespace requires no capability (since Linux 3.8), and a process becomes namespaced-root inside it; combined with a set-user-ID-root program whose UID matches the namespace creator's UID conferring capabilities, this is the mechanism by which in-container root regains apt/chown/chmod caps mapped to unprivileged host uids.
  — source: https://man7.org/linux/man-pages/man7/capabilities.7.html
- **[S6]** (vote 0-3) Firecracker issue #2142 tracks the capability to create incremental (diff) snapshots as a planned feature, not a completed implementation, distinct from full snapshots.
  — source: https://github.com/firecracker-microvm/firecracker/issues/2142
