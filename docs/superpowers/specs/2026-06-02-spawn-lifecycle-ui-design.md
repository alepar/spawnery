# Spawn Lifecycle UI Design

**Date:** 2026-06-02
**Status:** Approved (brainstorming)
**Epic:** sp-7k8 lineage — UI spawn lifecycle (new epic, see Beads section)

## Goal

Make spawns first-class, independently-named, multi-instance objects in the web UI:
multiple instances of the same app can run side-by-side; each spawn has its own
backend-persisted name (autogenerated, renamable); spawns can be suspended/resumed
and stopped from the UI; the sidebar shows each spawn's live status; and switching
back into a spawn replays its prior transcript.

## Decisions (locked during brainstorming)

1. **Name storage = backend-persisted.** A spawn carries an independent `name`
   (proto + store column + `RenameSpawn` RPC). `spawn_id` remains the real key;
   `name` is the display/headline label.
2. **Suspend = non-lossless, now.** `SuspendSpawn`/`ResumeSpawn` are wired for
   real this slice (they are `Unimplemented` stubs today). Suspend tears down the
   container but keeps the row (`suspended`); resume provisions a fresh container.
   In-container working state and agent memory are NOT preserved across suspend —
   *lossless* suspend stays gated on E3 persistent storage + node suspend plumbing
   (sp-gd9). This is a separate concern from transcript replay (below).
3. **Session model = one active session at a time.** A single live ws/client for
   the selected spawn. Background spawns keep running on the node but are
   disconnected. Switching spawns closes the current ws and (if the target is
   `active`) opens a new one.
4. **History on switch = adapter records + replays (visual transcript only).**
   The in-container `acpadapter` becomes an ACP-aware recording proxy and replays
   the transcript to any client that connects. Gives cross-device + reload-while-
   active visual continuity. NOT agent memory; LOST on suspend (container
   destroyed). Durable/true history + on-demand pagination = post-demo epics.
5. **Status dot in the sidebar.** Each spawn row shows a colored dot:
   green = `active`, grey = `suspended`, red = `errored`/`unreachable`,
   amber (pulsing) = transitional `starting`/`suspending`.

### Lifecycle action semantics

| UI action | Container | Row state | In list? | RPC |
|---|---|---|---|---|
| **Suspend** | torn down | `suspended` | yes | `SuspendSpawn` |
| **Resume** | fresh container | `active` | yes | `ResumeSpawn` |
| **Stop** | torn down | soft-deleted | no | `DeleteSpawn` |
| **Rename** | unchanged | unchanged | yes | `RenameSpawn` |

(`StopSpawn` legacy RPC is retained for spawnctl + e2e; the UI "Stop" maps to
`DeleteSpawn` for the soft-delete-and-drop-from-list semantics.)

## Slice 1 — Backend lifecycle (pure CP/store, hermetic)

### Proto (`proto/cp/v1/cp.proto`) — additive only

- `SpawnSummary` gains `string name = 8;`
- `CreateSpawnRequest` gains `string name = 6;` (optional; empty ⇒ server autogen)
- New RPC `rpc RenameSpawn(RenameSpawnRequest) returns (RenameSpawnResponse);`
  - `RenameSpawnRequest { string spawn_id = 1; string name = 2; }`
  - `RenameSpawnResponse {}`
- `SuspendSpawn`/`ResumeSpawn` already declared — implement them.

Regenerate with the pinned buf toolchain (buf@v1.45.0, protoc-gen-go@v1.34.2,
protoc-gen-connect-go@v1.16.2; `$(go env GOPATH)/bin` on PATH).

### Store (`internal/cp/store`) — migration 0004

- `spawns` gains `name TEXT NOT NULL DEFAULT ''`.
- **No DB uniqueness constraint on name.** Names are a display label; `spawn_id` is
  the real key, and manual rename explicitly allows duplicates — so a hard
  `UNIQUE(owner_id, name)` is wrong (it would reject a duplicate rename, and turn a
  rare concurrent-autogen collision into a 500 instead of a harmless duplicate
  label). Autogen dedup is therefore **app-side, best-effort** (query the owner's
  existing names, pick the next free suffix). A theoretical concurrent-create race
  may yield two equal labels; harmless, and `spawn_id` still disambiguates.
- Migration provides both sqlite and pg up/down; pg path covered by the existing
  build-tagged schema-soundness test (not a CI stack).

New repo methods (extend the existing guarded-transition pattern; reuse the
`Suspending`/`Suspended` statuses already in the enum):
- `Rename(ctx, id, name) error` — `UPDATE spawns SET name=? WHERE id=? AND status<>deleted`; `ErrNotFound` on 0 rows.
- Suspend path: `MarkSuspending(id, gen)` → (teardown) → `MarkSuspended(id, gen)`,
  status+generation guarded; ends the live container row.
- Resume path: `ClaimStarting` a new-generation container (end-old-then-insert-new,
  the same primitive `CreateSpawn` uses) → `SetActive`.
- Name generation helper (server-side, authoritative): given owner + desired base
  name, return the first free `base`, `base 2`, `base 3`, … among that owner's
  non-deleted spawn names.

### CP handlers

- `CreateSpawn` (`internal/cp/server.go`): if `req.Name == ""`, default the base
  name to the resolved app's display name (fallback to `app_id` if display name is
  empty), then run the dedup helper; persist `name` on the spawn row. A
  client-supplied non-empty `name` is used verbatim (still length-validated).
- `RenameSpawn` (`internal/cp/lifecycle.go`): owner-guarded (`OwnerFromContext`;
  `Unauthenticated` if absent), validate name non-empty and ≤ 80 runes, call
  `store.Rename`. Manual rename **allows duplicates** (spawn_id is the key);
  uniqueness only matters for autogen legibility. `NotFound` if the spawn isn't
  the caller's.
- `SuspendSpawn` (`internal/cp/lifecycle.go`): owner-guarded; under the per-spawn
  lock, transition `active → suspending`, run the **existing stop teardown** (same
  path `stop()` uses to tell the node to tear down the container), then
  `→ suspended`. No new node message — reuses the teardown that `StopSpawn`/
  `DeleteSpawn` already perform; the difference is the terminal status.
- `ResumeSpawn` (`internal/cp/lifecycle.go`): owner-guarded; under the per-spawn
  lock, transition `suspended → starting`, re-`Provision` from the row's retained
  `app_id`/`app_version`/`model`/`mounts` (the same scheduler path `CreateSpawn`
  uses, bumping generation), then `SetActive` (`SetError` on failure, mirroring
  `CreateSpawn`'s compensating cleanup). Returns `FailedPrecondition` if the spawn
  isn't `suspended`.
- `ListSpawns` (`internal/cp/lifecycle.go`): populate the new `Name` field.

### Slice 1 tests (hermetic, `:memory:` + `-race`)

- store: the `name` column round-trips and the store allows duplicate names (no
  uniqueness constraint — dedup is app-side); `Rename` updates + `ErrNotFound` on
  missing/deleted; suspend/resume transitions are status+generation guarded.
- CP: `RenameSpawn` owner guard + length cap + duplicate-allowed; `CreateSpawn`
  autogen + collision suffix; `SuspendSpawn` reaches `suspended` and tears down;
  `ResumeSpawn` reaches `active` with a bumped generation and re-provisions; resume
  of a non-suspended spawn → `FailedPrecondition`; `ListSpawns` returns names.
- migration 0004 sqlite applies; pg schema-soundness test compiles/loads.

## Slice 2 — Adapter history replay (node/adapter + web client)

### `deploy/agent/acpadapter` — ACP-aware recording proxy

The wire is newline-delimited JSON (matches `web/src/acp/conn.ts`). The adapter
keeps its current attach/detach + half-close + single-persistent-pump semantics
but parses each direction line-by-line and records a transcript.

- **agent→client pump** (`bridge.go`): replace the 32 KB byte read with a
  `bufio.Reader.ReadBytes('\n')` loop. For each line: forward the **exact bytes**
  to the current conn (lossless — preserves streaming + framing), then parse a
  copy; if `method == "session/update"`, fold into the transcript.
- **client→agent** (`serve`'s `io.Copy`): replace with the same line loop —
  forward each line verbatim to agent stdin, then if `method == "session/prompt"`,
  record a user item from `params.prompt[].text`.
- **Transcript** (mutex-guarded `[]item`, `item = {Role, Text, Title, Status}`)
  with the same coalescing the client does:
  - `session/prompt` ⇒ push a `user` item (joined text parts) + open a fresh
    `agent` turn target.
  - `agent_message_chunk` ⇒ append `content.text` to the current `agent` item.
  - `agent_thought_chunk` ⇒ append to the current `thought` item.
  - `tool_call` ⇒ push a `tool` item (`title`, `status`); `tool_call_update` ⇒
    update the matching tool item's status (by `toolCallId`).
  - Cap at 500 items; when exceeded, drop oldest and keep a single leading
    `{Role:"system", Text:"earlier history truncated"}` marker. Pagination/on-demand
    loading = post-demo epic.
- **Replay on attach** (`connHub.set`): when a new client attaches, before
  resuming forwarding, write one line to the new conn:
  `{"jsonrpc":"2.0","method":"spawn/history","params":{"items":[…]}}`.
  Out-of-band of the ACP request/response flow (no id correlation, no sessionId
  coupling). Snapshot the transcript under the mutex to build the frame.

### `web/src/acp/client.ts`

- Add `onHistory?: (items: HistoryItem[]) => void` to the client (settable like a
  handler, independent of `PromptHandlers`).
- `route`: add `if (m.method === "spawn/history") { this.onHistory?.(m.params?.items ?? []); return; }`.
- `HistoryItem` type mirrors the adapter item; a small mapper converts it to the
  UI `Item` shape.

Rationale for a custom method over replaying raw `session/update`: between prompts
the client's `handlers` is `{}`, so replayed `session/update`s would be dropped;
and replaying request/response pairs would feed the client response ids it never
issued. A dedicated notification handled in `route` always fires.

### Slice 2 tests

- adapter Go (hermetic): drive a scripted ndjson exchange (prompt + chunk + thought
  + tool_call updates) through the recording pump; a second client that dials gets
  a `spawn/history` frame whose items match the expected coalesced transcript;
  verbatim forwarding remains byte-exact (existing `bridge_test.go` cases stay
  green); cap/truncation marker appears past 500 items.
- client vitest: a fake ws delivering a `spawn/history` frame fires `onHistory` with
  mapped items; existing client tests unchanged.

## Slice 3 — Web multi-spawn UI

### `web/src/App.tsx` — multi-spawn state

Replace the single `activeApp`/`spawnRef` with:
```
spawns: Map<spawnId, { name, appId, status, items }>   // items = render/transcript buffer
activeId: string | null
wsRef / clientRef / genRef                              // ONE live session, for activeId
```
- **On mount + after every lifecycle action:** call `ListSpawns` to hydrate the map
  (names + statuses from the durable ledger). Light poll (~3s while mounted) keeps
  statuses fresh (suspend completing, `unreachable`, errors).
- **Create:** `CreateSpawn` (no name → server autogen) → add to map → make active →
  open ws.
- **Switch:** clicking a spawn closes the current ws (container keeps running),
  restores that spawn's `items`, and **only if the target is `active`** opens a new
  ws; `onHistory` then replaces that spawn's `items` with the authoritative replay.
  Selecting `suspended`/`errored` shows the buffer + a Resume affordance, no ws.
- **History wiring:** set `clientRef.onHistory` to replace the active spawn's items.
- Keep the `genRef` re-entrancy guard so stale ws callbacks don't clobber a newer
  switch.

### `web/src/shell/Sidebar.tsx` — spawn list

The "Spawns" section becomes a list. Each row:
```
● Wiki 2                ⋯      ← status dot + spawn name (headline)
  spawnery/wiki                ← app id (subline, muted)
```
- **Status dot** colors: green `active` · grey `suspended` · red `errored`/
  `unreachable` · amber (pulsing) `starting`/`suspending`. `data-testid="spawn-dot-<id>"`
  with a status class for test assertions.
- **`⋯` kebab menu** per row: **Rename**, **Suspend**/**Resume** (toggles by status),
  **Stop** (destructive, confirm). Rename also via double-click on the headline →
  inline edit → `RenameSpawn`.
- Active row highlighted; clicking the row body switches to its chat.
- Empty state: "— none yet —".

### `web/src/shell/AppShell.tsx`

`view` stays `chat|market|settings`; the chat view renders the `activeId` spawn's
items/status. Spawning from the marketplace **adds-and-activates** (no longer
replaces). Thread the spawns list + handlers (select/rename/suspend/resume/stop)
from App → AppShell → Sidebar.

### `web/src/api/spawnlet.ts`

Add `listSpawns()`, `renameSpawn(id,name)`, `suspendSpawn(id)`, `resumeSpawn(id)`.
Point the UI "Stop" at `DeleteSpawn` (add `deleteSpawn(id)`); keep the existing
`stopSpawn` (legacy `StopSpawn`) for non-UI callers.

### Slice 3 tests

- vitest: Sidebar renders a multi-row list; status dot class per status; kebab
  actions invoke the right API; double-click → inline rename → `renameSpawn`;
  App multi-spawn switch restores items and opens/closes ws appropriately; selecting
  a suspended spawn shows Resume and opens no ws.
- Playwright e2e: spawn two `secret-app` instances → two rows with distinct names
  (collision counter) + green dots; rename one (headline updates); suspend one (dot
  greyed, Resume appears) → resume (dot green); stop one (row drops); chat in A,
  switch to B then back to A → prior bubbles replayed.

## Post-demo epics (filed, not in scope)

- **Durable / true history:** agent memory across reconnect (ACP `session/load` if
  goose supports it) and/or CP/store transcript persistence surviving suspend and
  cross-device.
- **On-demand history pagination:** lazy/windowed loading for long transcripts;
  removes the adapter's 500-item cap.

## Non-goals

- Lossless suspend/resume (gated on E3 + sp-gd9).
- Concurrent live sessions (one active at a time by decision).
- A distinct "stopped-but-kept" state (suspend covers "keep it, off"; Stop removes).
